1. Hova esik a REST? – az API-stílusok térképe

Alapfogalom Taxonómia

Az interjún a „mesélj a REST-ről" kérdés a web API-k / API-paradigmák témakörbe esik. A REST egy architekturális stílus (nem protokoll, nem szabvány!) a kliens és a szerver közti kommunikációra. Hogy lásd, hova illik, előbb térképezzük fel a tájat.

Két fő tengely mentén érdemes gondolkodni:

  • Interakciós modell: kérés–válasz (a kliens kér, a szerver válaszol) vagy valós idejű / streaming (folyamatos, push).
  • Absztrakciós szint: erőforrás-orientált (REST), lekérdezés-orientált (GraphQL), vagy művelet/függvény-orientált (RPC, gRPC, SOAP).

A REST a „kérés–válasz, erőforrás-orientált" sarokban él — ez a web legelterjedtebb API-stílusa. Az alternatívák más absztrakciót (GraphQL, RPC) vagy más interakciós modellt (valós idejű) kínálnak.

Fontos megkülönböztetés: a REST/GraphQL/SOAP architekturális stílus vagy specifikáció, a HTTP/HTTP2 transport protokoll, a JSON/XML/Protobuf adatformátum. Egy interjún jó pont, ha külön kezeled ezeket: pl. „a REST jellemzően HTTP-n, JSON formátummal, de elvileg formátum- és (részben) transport-független".

Idempotencia (gyakori kérdés): egy művelet idempotens, ha ugyanazt a kérést kétszer (vagy N-szer) elküldve ugyanaz az eredmény, mint egyszer. A PUT és DELETE idempotens (kétszeri „töröld a 42-est" ugyanoda vezet), a POST nem (kétszeri „hozz létre" két rekordot csinál). Ezért fontos hálózati újrapróbálkozásnál.

HTTP státuszkódok – 5 család

A státuszkód a válasz „címkéje". Jó REST API a megfelelő kódot adja: 201 létrehozásnál, 404 ha nincs ilyen erőforrás, 401 ha nincs bejelentkezve, 403 ha nincs joga, 409 ütközésnél.

3. REST részletesen

REST Architekturális stílus

REST = REpresentational State Transfer. Roy Fielding doktori disszertációjában definiálta (2000). Nem szabvány, hanem architekturális stílus: megszorítások (constraints) gyűjteménye. A lényeg: a rendszert erőforrások (resources) köré szervezzük, amelyeket URL-ek azonosítanak, és HTTP-igékkel manipulálunk.

A REST 6 megszorítása (constraint)

MegszorításJelentés
Client–ServerA kliens (UI) és a szerver (adat) szétválik, függetlenül fejleszthető.
StatelessMinden kérés önmagában teljes — a szerver NEM tárol kliens-session-t a kérések között. (Skálázhatóság!)
CacheableA válaszok megjelölhetők cache-elhetőként (HTTP cache, ETag, CDN).
Uniform InterfaceEgységes felület: erőforrások URL-en, HTTP-igék, reprezentációk (JSON), hipermédia. Ez a REST szíve.
Layered SystemKözbülső rétegek (proxy, gateway, load balancer) beékelhetők átláthatóan.
Code on Demand (opcionális)A szerver futtatható kódot is küldhet (pl. JS). Az egyetlen opcionális megszorítás.
Csapda: a legtöbb „REST API" valójában nem teljesen RESTful — ritka a hipermédia (HATEOAS). Iparági szóhasználatban a „REST API" gyakran csak „HTTP+JSON, erőforrás-URL-ek, HTTP-igék" (lásd Richardson Level 2 lent). Interjún jó pont, ha tudod ezt a különbséget.

Richardson Maturity Model – „mennyire RESTful?"

Leonard Richardson modellje (Martin Fowler népszerűsítette) 4 szinten méri, mennyire kihasznált a web. A „valódi" REST a 3. szint.

Level 0: minden egy endpointra megy (mint az RPC). Level 1: külön URL minden erőforrásnak. Level 2: a HTTP-igék és státuszkódok helyes használata (a gyakorlati „REST"). Level 3: a válasz hivatkozásokat (linkeket) ad a következő lehetséges műveletekhez — ez a HATEOAS.

Erőforrás → URL tervezés (a gyakorlatban)

A REST URL-ek főneveket használnak (erőforrások), nem igéket. Az igét a HTTP-metódus adja.

GET    /users            → összes felhasználó (lista, lapozható)
POST   /users            → új felhasználó létrehozása
GET    /users/42         → a 42-es felhasználó
PUT    /users/42         → a 42-es teljes cseréje
PATCH  /users/42         → a 42-es részleges módosítása
DELETE /users/42         → a 42-es törlése
GET    /users/42/orders  → a 42-es felhasználó rendelései (beágyazott erőforrás)

# ❌ ROSSZ (ige az URL-ben, RPC-stílus):
POST /createUser     GET /getUser?id=42     POST /deleteUser

# ✅ Lapozás, szűrés, rendezés query paraméterrel:
GET /users?page=2&limit=20&role=admin&sort=-createdAt

Példa REST kérés + válasz

# Kérés
GET /users/42 HTTP/1.1
Accept: application/json

# Válasz
HTTP/1.1 200 OK
Content-Type: application/json
ETag: "a1b2c3"
Cache-Control: max-age=60

{
  "id": 42,
  "name": "Anna",
  "email": "anna@pelda.hu"
}

HATEOAS (Level 3) – mit jelent?

A válasz tartalmazza a következő lehetséges műveletek linkjeit, így a kliens nem „égeti be" az URL-eket — a szerver vezeti. Ritkán használt, de interjún ismerni kell.

{
  "id": 42,
  "name": "Anna",
  "status": "active",
  "_links": {
    "self":       { "href": "/users/42" },
    "orders":     { "href": "/users/42/orders" },
    "deactivate": { "href": "/users/42/deactivate", "method": "POST" }
  }
}

4. GraphQL

GraphQL Query language

A GraphQL (Facebook, 2012, 2015-ben nyílt forrású) egy lekérdezőnyelv az API-khoz. Egyetlen endpoint (POST /graphql), és a kliens pontosan megmondja, milyen mezőket kér — se többet, se kevesebbet. A REST két fő fájdalmára válasz: over-fetching és under-fetching.

REST: a profil-oldalhoz több endpointot kell hívni, és mindegyik a teljes objektumot adja. GraphQL: egyetlen query pontosan a kívánt mezőket kéri le, egy kérésben.

Séma, query, mutation

# --- Séma (SDL) – a szerver definiálja a típusokat ---
type User {
  id: ID!
  name: String!
  posts: [Post!]!
}
type Post { id: ID!  title: String! }

type Query    { user(id: ID!): User }
type Mutation { addPost(userId: ID!, title: String!): Post }
# --- A kliens query-je: pontosan a kért mezők ---
query {
  user(id: 42) {
    name
    posts { title }      # csak a title kell, a body nem
  }
}

# --- Válasz: a query alakját tükrözi ---
{ "data": { "user": { "name": "Anna",
  "posts": [ { "title": "Hello" }, { "title": "GraphQL" } ] } } }
GraphQL buktatók:
  • N+1 probléma: a resolverek naivan külön lekérdezést futtatnak minden elemre → DataLoader (batch + cache) a megoldás.
  • HTTP-cache nehéz: minden POST /graphql, így a böngésző/CDN GET-cache nem működik (persisted queries / kliens-cache, pl. Apollo kell).
  • Komplexitás-támadás: mélyen beágyazott query lefárasztja a szervert → query depth/complexity limit kell.
Subscriptions: a GraphQL valós idejű frissítést is tud (subscription), jellemzően WebSocket felett — így a „realtime" igény is lefedhető vele.

5. gRPC

gRPC RPC HTTP/2

A gRPC (Google) egy nagy teljesítményű RPC-keretrendszer. HTTP/2-n fut, az adatformátum bináris Protocol Buffers (protobuf). Contract-first: egy .proto fájlban definiálod a szolgáltatást és az üzeneteket, majd kód generálódik belőle (kliens + szerver stubok) sok nyelvre.

// user.proto  — a szerződés (contract)
syntax = "proto3";

message GetUserRequest { int32 id = 1; }
message User { int32 id = 1; string name = 2; string email = 3; }

service UserService {
  rpc GetUser (GetUserRequest) returns (User);
  rpc ListUsers (Empty) returns (stream User);   // server streaming
}
// → protoc generál: kliens stub + szerver interfész (Go, Java, TS, …)

A 4 kommunikációs mód

A gRPC a HTTP/2 multiplexálását kihasználva 4 streaming módot kínál — ezért ideális mikroszolgáltatások közti, nagy átviteli sebességű kommunikációra.

ElőnyökHátrányok
Nagyon gyors (bináris protobuf, HTTP/2)Böngészőből nem közvetlen (grpc-web proxy kell)
Erős típusosság, kódgenerálás, contract-firstNem ember-olvasható (bináris) → nehezebb debug
Beépített streaming (4 mód)Kevésbé „böngésző-natív", mint a REST
Polyglot (sok nyelv)Tanulási görbe, tooling-igény
Tipikus használat: belső mikroszolgáltatások közti kommunikáció (kelet–nyugati forgalom). Publikus, böngészős API-nak inkább REST/GraphQL.

6. SOAP

SOAP XML Enterprise

A SOAP (Simple Object Access Protocol) egy protokoll (nem stílus!), XML-alapú üzenetekkel. Szigorú szerződés a WSDL (Web Services Description Language). Transport-független (HTTP, de akár SMTP is). A WS-* szabványok (WS-Security, WS-AtomicTransaction) miatt erős a vállalati/banki/legacy világban.

Minden SOAP-üzenet egy „borítékba" (Envelope) van csomagolva: opcionális Header + kötelező Body, hiba esetén szabványos Fault elem.

<soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope">
  <soap:Header/>
  <soap:Body>
    <getUser><id>42</id></getUser>
  </soap:Body>
</soap:Envelope>
Mikor találkozol vele? Régi/vállalati rendszerek, banki, biztosítói, állami integrációk, fizetési átjárók. Új zöldmezős projekthez ma már ritkán választják — nehézkes, bőbeszédű (XML), de a beépített biztonsági/tranzakciós szabványok miatt bizonyos helyeken máig él.

7. RPC, JSON-RPC, XML-RPC, tRPC

RPC tRPC

Az RPC (Remote Procedure Call) a legrégebbi ötlet: hívj meg egy függvényt/metódust a szerveren, mintha lokális lenne. Itt nem erőforrásokban, hanem műveletekben gondolkodsz.

JSON-RPC

# Kérés (POST /rpc)
{ "jsonrpc": "2.0", "method": "getUser", "params": { "id": 42 }, "id": 1 }
# Válasz
{ "jsonrpc": "2.0", "result": { "id": 42, "name": "Anna" }, "id": 1 }

tRPC – TypeScript end-to-end típusbiztonság

Modern, TS-only megoldás: a szerver eljárásait közvetlenül hívod a kliensből, kódgenerálás nélkül, teljes típus-átöröklődéssel (a backend típusai automatikusan elérhetők a frontenden). Ideális full-stack TS monorepóban (pl. Next.js).

// szerver
const appRouter = router({
  getUser: publicProcedure.input(z.object({ id: z.number() }))
    .query(({ input }) => db.user.find(input.id)),
});
export type AppRouter = typeof appRouter;   // ← csak a TÍPUS megy a kliensre

// kliens — teljes autocomplete + típusellenőrzés, build-időben:
const user = await trpc.getUser.query({ id: 42 });
REST vs RPC szemlélet: REST = „milyen dolgokon (erőforrásokon) dolgozom?" (főnevek + HTTP-igék). RPC = „milyen műveletet hívok meg?" (igék/metódusnevek). A gRPC és a SOAP is RPC-szemléletű.

8. Valós idejű: Polling, SSE, WebSocket, Webhook

Realtime Streaming

A sima kérés–válasz nem alkalmas arra, hogy a szerver kezdeményezzen (pl. chat-üzenet, élő ár). Négy megközelítés:

Long polling: ismételt kérések. SSE: egy nyitott HTTP-kapcsolaton a szerver folyamatosan push-ol (egyirányú). WebSocket: tartós, kétirányú csatorna. Webhook: a külső szerver hív téged egy esemény hatására (fordított API, szerver–szerver).

IrányKapcsolatTipikus használat
Long pollingkliens kérismételt HTTPfallback, ha nincs jobb
SSEszerver → kliens1 tartós HTTPértesítés, élő feed, dashboard
WebSocketkétirányútartós (ws://)chat, játék, együttműködés
Webhookszerver → szervereseti HTTP POSTfizetés (Stripe), CI, integráció

10. Mikor melyiket? – döntési fa

Döntés Gyakorlat

Hüvelykujjszabály: publikus/egyszerű → REST; sokféle kliens, túl sok/kevés adat → GraphQL; belső, nagy teljesítményű → gRPC; valós idejű → WebSocket/SSE/Webhook; vállalati/legacy → SOAP.

A gyakorlatban gyakran keverék: publikus REST API + belső gRPC a service-ek közt + WebSocket a realtime funkcióhoz + Webhook a 3rd-party integrációkhoz (Stripe, GitHub). Egy rendszer több stílust is használ, a feladathoz illeszkedve.

11. CQRS – kapcsolódó minta (más kategória!)

Architektúra Kapcsolódó
Miért nem volt a stílus-térképen? Mert a CQRS nem API-kommunikációs stílus (mint a REST/GraphQL/gRPC). Az API-stílusok azt írják le, hogyan beszél a kliens és a szerver; a CQRS egy backend architekturális minta arról, hogyan szervezed a szerveren belül az írás- és olvasás-logikát. Más szinten van — de szorosan kapcsolódik az API-tervezéshez, és interjún simán előjöhet az API-k mellett, ezért érdemes ismerni.

CQRS = Command Query Responsibility Segregation. A lényeg: a parancsokat (Command — írás/módosítás) és a lekérdezéseket (Query — olvasás) külön útvonalon, gyakran külön modellel (akár külön adattárral) kezeled. A CQS-elv (Bertrand Meyer) skálázása architektúra-szintre: egy művelet vagy módosít (command, nincs visszaadott adat), vagy lekérdez (query, nincs mellékhatás) — de nem mindkettő.

A parancs- (írás) és lekérdezés- (olvasás) oldal külön modell/adattár. A write store változásai „vetülnek" a read store-ba (gyakran eseményekkel) — ezért a read oldal kicsit késhet (eventual consistency).

Miért csinálnánk ezt?

  • Eltérő terhelés: tipikusan sokkal több az olvasás, mint az írás → a két oldalt külön skálázhatod (pl. sok read replika).
  • Eltérő modell: az íráshoz normalizált, konzisztens modell jó; az olvasáshoz denormalizált, a nézethez szabott „read model" gyorsabb.
  • Tisztább felelősség: a parancs-logika (validáció, üzleti szabályok) elválik a lekérdezéstől.

Hogyan kapcsolódik az API-khoz?

# REST: a parancs és a lekérdezés szétválik (akár task-alapú végpontok)
POST   /orders               # Command – létrehoz (írás)
POST   /orders/42/cancel     # Command – task-alapú művelet (nem nyers CRUD!)
GET    /orders/42            # Query  – olvasás (akár külön, optimalizált read modellből)

# GraphQL: az API szinten BEÉPÍTETT command/query szétválasztás:
query    { order(id: 42) { status } }        # olvasás (Query)
mutation { cancelOrder(id: 42) { status } }  # parancs (Command)

# HTTP-szinten a CQS gondolat már ott van: GET = biztonságos lekérdezés,
# POST/PUT/DELETE = mellékhatásos parancs.
CQRS és Event Sourcing: gyakori társ, de NEM ugyanaz. Az Event Sourcing az állapotot események sorozataként tárolja (nem a pillanatnyi állapotot), és a read modellek ezekből az eseményekből „vetülnek". A CQRS működik Event Sourcing nélkül is (pl. csak külön read/write DB), és fordítva.
Mikor NE használd? A CQRS komplexitást ad (két modell, szinkronizáció, eventual consistency a read oldalon). Egyszerű CRUD-alkalmazásnál overkill — ott egy modell + REST bőven elég. Csak akkor éri meg, ha a read és write terhelése/modellje tényleg jelentősen eltér (pl. nagy forgalmú, esemény-vezérelt rendszer).
Egymondatos összefoglaló (interjúra): a CQRS nem API-stílus, hanem backend-minta: az írást (command) és az olvasást (query) külön modellel/úton kezeli — a GraphQL query/mutation és a REST-ben a GET vs POST/PUT/DELETE pont ezt a szétválasztást tükrözi az API szintjén.

12. Interjúkérdések és válaszvázak

Interjú
K1: Mesélj a REST-ről.

A REST egy architekturális stílus (Roy Fielding, 2000) HTTP-alapú API-khoz. Erőforrásokban gondolkodik, amiket URL azonosít, és HTTP-igékkel (GET/POST/PUT/PATCH/DELETE) manipulál. Kulcsfogalmak: stateless (a szerver nem tárol session-t), uniform interface, cacheable (GET + ETag). A státuszkódok jelzik az eredményt (200/201/404/401…). 6 megszorítása van; a HATEOAS (hipermédia) a „teljes" REST, de a gyakorlatban ritka.

K2: REST vs GraphQL?

REST: sok endpoint, a szerver dönti el a válasz alakját → over/under-fetching, de kiváló HTTP-cache. GraphQL: egy endpoint, a kliens kéri pontosan a mezőket → nincs over-fetching, egy körutazás, de nehezebb cache (POST) és N+1/komplexitás veszély. REST egyszerű CRUD-ra, GraphQL sokféle kliensre és összetett/aggregált adatra jó.

K3: Mi az idempotencia, és melyik metódus idempotens?

Idempotens: N-szer elküldve ugyanaz az eredmény, mint egyszer. GET, PUT, DELETE, HEAD, OPTIONS idempotens; POST és (általában) PATCH nem. Hálózati újrapróbálkozásnál (retry) kritikus — POST-nál idempotency key-t használunk.

K4: REST vs RPC (pl. gRPC) szemlélet?

REST = erőforrás-orientált (főnevek + HTTP-igék: GET /users/42). RPC = művelet-orientált (metódushívás: getUser(42)). A gRPC RPC-stílusú, HTTP/2 + protobuf, nagyon gyors, belső service-ekhez ideális, de böngészőből proxy kell.

K5: Mikor választanál WebSocketet REST helyett?

Ha valós idejű, kétirányú kommunikáció kell (chat, élő játék, kollaboráció). Ha csak a szerver push-ol egy irányba → SSE elég (egyszerűbb, auto-reconnect, HTTP). Ha 3rd-party rendszer értesít → Webhook. A REST továbbra is jó a sima CRUD-műveletekre ugyanabban az appban.

Interjú tipp: ne mondd, hogy „a GraphQL/gRPC mindig jobb a REST-nél". Mindig kontextus-függő: a REST egyszerűsége, cache-elhetősége és univerzális toolingja sok esetben verhetetlen. A jó válasz a trade-offokat mutatja be, és a feladathoz illeszti a választást.
