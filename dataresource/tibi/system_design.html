1. CAP háromszög (CAP Theorem)

Rendszertervezés Nagyszabású szisztémák Elosztott rendszerek

Mit jelent a CAP?

A CAP theorem (Brewer-tétel) azt állítja, hogy egy elosztott rendszerben egyszerre csak két tulajdonság teljesülhet:

Elosztott rendszerben egyszerre csak két tulajdonság garantálható — minden él egy választás: CP, AP vagy CA.

A három oszlop részletesen

1. Consistency (Konzisztencia)

A rendszer összes csomópontja ugyanazt az adatot látja ugyanabban az időben. Ha módosítasz egy adatot, az összes olvasó azonnal az új értéket kapja.

  • Erős konzisztencia (Strong Consistency): Szinkron frissítés az összes replika között
  • Gyenge konzisztencia (Eventual Consistency): Idővel konzisztens lesz
  • Monoton konzisztencia (Monotonic Consistency): Egyenlő vagy újabb verzió garantált
User A: PUT /user/1 { name: "John" }
              ↓ (Consistency: azonnal frissül minden szervernél)
User B: GET /user/1 → { name: "John" }  ✅ azonnal látja az új értéket

2. Availability (Elérhetőség)

A rendszer minden kérésre választ ad, és nem dől össze. Még ha egy csomópont leáll is, az összes többi működik.

  • Nem lehet "kérés időtúllépés" vagy "szerver elérhetetlen" válasz
  • Az 99.99% uptime (4 nines)
  • Ha egy szerver leáll, a maradék rendszer továbbra is válaszol
Server 1 (EU): leáll
    ↓
User (EU): GET /data → Server 2 (EU) → 200 OK  ✅ válasz érkezik

3. Partition Tolerance (Partíció Tolerancia)

A rendszer akkor is működik, ha a csomópontok között a hálózat szétszakad (partition). Az egyik csoport nem tudja elérni a másikat.

  • Hálózati késés vagy szétszakadás nem állítja meg a rendszert
  • Elosztott rendszereknél MINDIG szükséges (nem választható)
Network Partition:
[DC1: Server A] ═══ OFFLINE ═══ [DC2: Server B]

A továbbra is szolgálja az EU felhasználókat
B továbbra is szolgálja az US felhasználókat
    ↓ A kettő között nincs szinkronizáció, de mindkettő működik

CAP választások: CP, AP, CA

Típus Tulajdonságok Mit áldozunk fel Példák
CP Consistency + Partition Tolerance ❌ Availability (szerver válhat elérhetetlenné) MongoDB, PostgreSQL, HBase, Kafka
AP Availability + Partition Tolerance ❌ Consistency (adatok lehetnek inkonzisztensek) Cassandra, DynamoDB, Riak, CouchDB
CA Consistency + Availability ❌ Partition Tolerance (hálózati szétszakadást nem tűri) Tradicionális RDBMS (nem elosztott), MySQL, Oracle

Döntési fa: Melyiket válassza?

1. MINDIG van Partition Tolerance szükség → CA ki van zárva
   (Hacsak nem monolitikus, egyszeres szerverről beszélünk)

2. Konzisztencia vs Elérhetőség választás:

   CP választás (Konzisztencia > Elérhetőség):
   ✅ Pénzügyi rendszerek (bank transzferek: konzisztencia kritikus)
   ✅ E-commerce rendelések (dupla vásárlás tilos)
   ✅ Zárolt erőforrások (csak egy felhasználó módosíthat egyszerre)
   ❌ De: egy hálózati partition alatt a rendszer válhat "read-only"-vá

   AP választás (Elérhetőség > Konzisztencia):
   ✅ Közösségi média (like-ok, Follow-ok: inkonzisztencia ok)
   ✅ Caching (régi adat is jó, mint üres válasz)
   ✅ Real-time analytics (szállítás követés: közelítő adat ok)
   ❌ De: időnként az adatok inkonzisztensek lesznek

Praktikus megközelítés: PACELC

A CAP theorem egy kiterjesztése a PACELC (Partition, Availability, Consistency, Else, Latency, Consistency):

IF Partition happens
  THEN choose Availability OR Consistency
ELSE (no partition)
  THEN choose Latency OR Consistency

Azaz: még nincs partition-ben (hálózat működik), lehet egy másik trade-off:
- Gyors válasz (alacsony latency) vagy
- Szinkron konzisztencia?
Modern megközelítés: A legtöbb nagy rendszer nem "tiszta" CAP választást követ, hanem:
  • Write-path: AP (felhasználó írásai azonnal mentésre kerülnek, később szinkronizálnak)
  • Read-path: Eventually Consistent (régi adatot olvashat, idővel frissül)
  • Critical operations: CP (pénzmozgás, zárolt erőforrás)
Ez a hibrid megközelítés a gyakorlatban a legáltalánosabb.

2. Spotify System Design – Zeneszállítási Platform

Rendszertervezés Média streaming Nagyszabású

Követelmények (Requirements)

Funkcionális követelmények:
  • Zenék streamelése online és offline módban
  • Zeneszerzők, albumok, lejátszási listák keresése és böngészése
  • Lejátszási listák létrehozása és megosztása
  • Ajánlási rendszer (hasonló zenék, személyre szabott javaslatok)
  • Lejátszási előzmények és statisztikák nyomon követése
  • Offline letöltés és szinkronizáció
Nem funkcionális követelmények:
  • Skálázás: 500M+ felhasználó, 100M+ dal
  • Latency: <200ms zeneszerzés indítása, <500ms keresés válasz
  • Elérhetőség: 99.9% uptime
  • Jitterless playback: Folyamatos lejátszás, nincs szaggatottság
  • P2P támogatás: Sávszélesség takarékosság peer-to-peer kereskedéssel

Kapacitás tervezés (Capacity Planning)

Feltételezések:
- 500M havi aktív felhasználó (MAU)
- Ebből 100M napi aktív (DAU)
- Átlag 30 óra lejátszás/hó/felhasználó
- Átlag bitrate: 128 kbps (adaptive: 32-320 kbps)

Számítások:
- Napi lejátszási idő: 100M DAU × 1 óra = 100M óra
- Napi szükséges sávszélesség: 100M × 3600s × 128kbps = 46.4 Petabit/nap
- Server: ~46.4 Pbps / 100 Gbps per server = ~464 szerver (streaming csak)
- Storage: 100M dal × 10 MB átlag = 1 PB + replikáció = 10 PB

Komponensek részletesen

1. CDN (Content Delivery Network) - Akamai

A zenék nagy része a szélső szerverekről kerül le (edge servers), nem a központi data centerekből.

  • Edge caching: Népszerű dalok az összes CDN szélén (US, EU, APAC)
  • Adaptive bitrate: Az eszköz és sávszélesség alapján automatikus minőség választás
  • Tartalom azonosítás: Perceptual hashing (ugyanaz a dal = ugyanaz a hash, akkor is máshogy kódolva)

2. Search & Discovery - Elasticsearch

Gyors keresés: "Drake", "Hip-hop 90s", "Napközentes zenék"

  • Elasticsearch cluster: 100M+ dal indexelve, full-text search
  • Faceting: Szűrés műfaj, évszám, előadó szerint
  • Suggestions: Auto-complete: "Drak..." → ["Drake", "Drake & Josh Theme"]
  • Replication factor: 3 (AP stratégia, eventual consistency)

3. Metadata Database - PostgreSQL

Strukturált adatok: felhasználó profil, lejátszási listák, követési információ

  • Felhasználó: ID, név, email, premium status, offline playlists
  • Dal: ID, cím, előadó, album, bitrate variáció, CDN URL
  • Lejátszási lista: Tulajdonos, dalok lista, módosítási timestamp, nyilvános/privát
  • Lejátszási előzmények: User-id, dal-id, timestamp, lejátszási idő (progress)
  • Replication: Master-Slave (CP stratégia)

4. Streaming Servers – Adaptive Bitrate Streaming

Valós idejű zeneszállítás HTTP vagy DASH/HLS protokollon

  • HTTP Progressive Download: Az egész fájl letöltésig lejátszik az elejét
  • DASH (Dynamic Adaptive Streaming over HTTP): Sávszélesség alapján automatikus bitrate váltás
  • Buffer management: 10-30 másodperces buffer az intézkedéshez (jitterless playback)
  • Session affinity: Ugyanaz az előadáshoz tartozó felhasználó ugyanazzal a szerverrel kommunikál (upload/pause állapot megtartása)

5. Storage – S3 / Google Cloud Storage

A tényleges hangfájlok tárhelye

  • Multi-region replication: Minden dal 3+ régióban replikálva (AP stratégia)
  • Naming: s3://spotify-audio/song-{id}/{bitrate}-{codec}.m4a
  • Lifecycle policy: Népszerűtlen dalok archiválása, de továbbra is elérhető

6. Cache - Redis

Gyors hozzáférés gyakran használt adatokhoz

  • Session cache: Aktuális lejátszási állapot (szüneteltethető, felmegy, előzmények)
  • Lejátszási lista cache: Top 1000 nyilvános lejátszási lista
  • User preferences: Kedvelt zene, követés lista
  • TTL: 1 óra tipikus, de ajánlások 24 óra

7. Recommendation Engine

Személyre szabott zeneajánlások (ML-alapú)

  • Collaborative Filtering: "Aki tetszett neked Drake, annak tetszett Kanye West"
  • Content-Based Filtering: "Ez a dal hip-hop, 2010s, urban → hasonló dalokat ajánlunk"
  • Context-aware: Délelőtt: energikus zene, éjszaka: relaxáló zene
  • Batch computing: Napi ML modell frissítés (Apache Spark)

Adatfolyam: Hogyan játszik le egy dal?

1. Felhasználó megnyomja a "Play" gombra
   ↓
2. iOS app → Spotify API: POST /player/play {track_id: "123"}
   ↓
3. API Gateway: Auth check (OAuth token)
   ↓
4. Play Service: User permission check, track availability
   ↓
5. Metadata DB: Lekérdezi, hogy hol van a track (CDN URL)
   ↓
6. Cache (Redis): Ellenőrzi, hogy ez a felhasználó már letöltötte-e offline
   ↓

   ↓

   ↓
9. Playback begins: iOS audio player indít
   ↓
10. Analytics Event: ga("send", {event: "track_play", track_id, user_id, timestamp})

Offline szinkronizáció

Offline mód (pl. repülőn):
1. Felhasználó előre letöltött 100 dalt: "Downloaded" státusszal a cache-ben
   ↓
2. Local SQLite DB: Lejátszási lista + dal metadata
   ↓
3. Offline lejátszás: sem szerver, sem internet szükséges
   ↓
4. WiFi visszakerülve: Upload lejátszási előzmények
   - Van-e szinkronizálandó lejátszási előzmény?
   - Sync: {user_id, tracks_played: [{id, timestamp, progress}]}
   ↓
5. Server: Frissítés analitikában, ajánlások frissítése

3. URL Shortener System Design – bit.ly-szerű szolgáltatás

Rendszertervezés Mapping Service Skálázható

Követelmények (Requirements)

Funkcionális követelmények:
  • Hosszú URL → rövid URL konverzió (pl. https://example.com/very/long/path → tinyurl.com/abc123)
  • Rövid URL → hosszú URL átirányítás (HTTP 301 vagy 302)
  • Opcionális: Custom shortcode (felhasználó dönthet: tinyurl.com/mylink)
  • Link lejárati idő (TTL) - pl. 1 hó, 1 év, soha nem jár le
  • Statisztikák nyomon követése (kattások, forrás, időbélyeg)
  • Felhasználó számára link kezelés (megtekintés, törlés, szerkesztés)
Nem funkcionális követelmények:
  • Skálázás: 10B+ rövidítés/nap, 100K+ redirect/másodperc csúcsidőben
  • Latency: <100ms redirect válasz
  • Elérhetőség: 99.99% uptime (redirect kritikus!)
  • Tárhelyi hatékonyság: 1T+ URL pár eltárolása
  • Nem szükséges: Szerkesztés (rövidítés után nem lehet módosítani)

Kapacitás tervezés (Capacity Planning)

Feltételezések:
- 10B új rövidítés/nap
- Read-Write ratio: 100:1 (100x több redirect, mint új rövidítés)
- 10B reads/nap = 116K reads/másodperc
- 100M writes/nap = 1.2K writes/másodperc

Tárhelyre:
- Évenkénti 3.6T rövidítés (10B × 365)
- Egy rövidítés ~500 bájt (URL, metaadat, statisztika)
- 1 év = 1.8 PB adat
- 5 év = 9 PB adat

Shortcode hossz:
- Base 62: [0-9a-zA-Z] = 62 karakter
- 6 karakter: 62^6 = 56.8 milliárd lehetőség ✅ (10B alá jól elfér)
- 7 karakter: 62^7 = 3.5 trillió lehetőség (overkill)

1. URL Shortening Logic – Algoritmus

Egy hosszú URL rövidítésének többféle módja:

Opció A: Hash-alapú (MD5 + Base62 konverzió) 
1. Hosszú URL: https://example.com/very/long/path?id=12345
2. MD5 hash: 2d3fbe4b5a2a5b7e8c9f0a1b2c3d4e5f (128 bit)
3. Truncate: első 6 karakter az MD5-ből: 2d3fbe
4. Base62 konverzió: abc123
5. Rövid URL: tinyurl.com/abc123

Probléma: Ütközések (hash collision)
- Bár ritka, lehetséges, hogy két URL-ből ugyanaz a rövidítés jön ki
Opció B: Database Counter (Guaranteed Unique) 
1. Hosszú URL: https://example.com/very/long/path?id=12345
2. Atomi lekérdezés: counter += 1 → 1000000001
3. Base62 konverzió: 1000000001 → "gjdgxs"
4. Rövid URL: tinyurl.com/gjdgxs

Előnyök: Garantáltan egyedi, szekvenciális
Hátrányok: Szekvenciális = könnyebb találgatni a URL-eket
Opció C: Distributed ID Generator (Snowflake-szerű) 
1. Snowflake ID generálása:
   - Timestamp (41 bit) | Data Center ID (5 bit) | Worker ID (5 bit) | Sequence (12 bit)
   - Teljes: 64 bit
2. Base62 konverzió: 64 bit → 11 karakter
3. Rövid URL: tinyurl.com/{11-char-base62}

Előnyök: Nincs DB lekérdezés, nem szekvenciális, skálázható
Ajánlott: Opció B vagy C (garantált egyediség)

3. Write Path – Rövidítés létrehozása

1. Kliens: POST /api/shorten
   {
     "original_url": "https://example.com/very/long/path?id=12345",
     "custom_shortcode": "mylink",  // opcionális
     "expiry_days": 30
   }

2. API Gateway: Load balancing, request validation
   
3. Write Service:
   a) Cache check: Van-e már ez az URL eltárolva?
      GET cache:url_hash("https://example.com/...") → cache hit?
   
   b) Ha cache miss:
      - Shortcode generálás (Snowflake vagy DB counter)
      - Custom shortcode validáció (még szabad-e?)
   
   c) DB INSERT:
      INSERT INTO url_mapping 
      (shortcode, original_url, user_id, created_at, expiry_at)
      VALUES ('abc123', 'https://example.com/...', 123, NOW(), NOW()+30days)
   
   d) Cache-be írás:
      SET cache:"abc123" "https://example.com/..." EX 86400

4. Válasz: 
   {
     "shortcode": "abc123",
     "short_url": "https://tinyurl.com/abc123",
     "original_url": "https://example.com/...",
     "created_at": "2026-02-04T10:00:00Z"
   }

4. Read Path – Átirányítás

1. Kliens: GET /abc123 (böngészőből)

2. Edge Cache (CloudFlare):
   Ellenőrzés: Van-e már gyorsítótárban?
   Cache hit? Azonnal HTTP 301 válasz
   Cache miss? → Origin szerverhez

3. Redirect Service (Read):
   a) Redis cache check:
      GET "abc123" → cache hit?
      Cache hit: 1-2 ms ✅
   
   b) Ha cache miss:
      SELECT original_url FROM url_mapping WHERE shortcode='abc123'
      5-10 ms ✅

   c) Ellenőrzés: Van-e lejárva?
      IF expiry_at < NOW() THEN return 404 Gone
   
   d) TTL frissítés a cacheban:
      SET "abc123" original_url EX 3600  (1 óra cache)

   e) Analytics event (async):
      PUBLISH kafka-topic:url_clicks {shortcode, ip, user_agent, timestamp}

4. Válasz:
   HTTP 301 Moved Permanently
   Location: https://example.com/very/long/path?id=12345
   Cache-Control: public, max-age=86400

Teljes latency: Redis hit = 2ms, DB hit = 10ms ✅ (célszám: <100ms)

5. Cache stratégia

Problem: 1T+ URL = nem fér be egy Redis instance memóriájába (tipikus: 100 GB)

Megoldás: Selective caching (hot URLs csak)

1. Write-aside cache:
   - Új rövidítés: automatikus cache-be (friss = hot)
   - TTL: 1 óra (vagy attól függően, mikor utolsó kattintás)

2. Read-through cache:
   - Olvasás a cache-ből
   - Ha miss: DB-ből, majd cache-be

3. LRU eviction:
   - Redis memory policy: allkeys-lru (legrégebben használt törlése)
   - Max memory: 100 GB per Redis instance
   - Túl sok URL? Horizontális skálázás (Redis Cluster)

4. Analytics bypass (nem cache-elendő):
   - Analytics adatok száraznak Kafkára (offline feldolgozás)
   - Ez nem készteti fel a Redis-t

│Replica 1   │          │Replica 2    │
│ 116K/sec   │          │ 116K/sec    │
└────────────┘          └─────────────┘

Sharding (gigantikus adatmennyiség):
Shortcode alapján sharding: hash(shortcode) % 10 = shard_id
- Shard 0: shortcode-ok "aaa"-"ccc"
- Shard 1: shortcode-ok "ccd"-"eee"
- ...
- 10 shard × 10 PB = 100 PB tárhelyre bővíthető

Sharding routing:
Client → API Gateway:
  shard_id = hash(shortcode) % num_shards
  route to: DB Cluster [shard_id]

CAP Analízis – URL Shortener

URL Shortener CAP stratégia: AP (Availability + Partition Tolerance)
Komponens Stratégia Indoklás
Redirect (Read) AP Mindig válaszolni kell, cache fallback, eventual consistency ok
Shortening (Write) AP Asszinkron DB replikáció, nem szinkron konzisztencia
Analytics AP Offline feldolgozás, nem real-time szükséges
Tárhelyre: Dupla shortcode CP Szükséges konzisztencia (nem azonos shortcode 2x)
Miért AP? Egy URL redirect szolgáltatásnál az elérhetőség kritikus. Ha a DB leáll, a cache-ből szolgálunk (eventual consistency: néhány perc késés az új rövidítésben ok).

Skálázás - 100K+ redirect/sec?

1. Horizontális skálázás:
   - 10 Redirect Service instance, Load Balancer mögött
   - Minden instance 10K req/sec kapacitása
   - 100K req/sec = 10 instance ✅

2. Redis Cluster:
   - Particionálva shortcode alapján
   - Read replicas: Pool 100 redis instance
   - Latency: <5ms garantált

3. Database Read Replicas:
   - 10 PostgreSQL Replica
   - Read load balanced
   - Cache hit rate: 90% (csak 10% DB hit)

4. CDN (CloudFlare):
   - Edge caching: népszerű shortcode-ok az összes szélén
   - Globális: US, EU, APAC, India
   - Latency: <50ms bárholról
