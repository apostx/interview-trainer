1. Érvénytelen JS data type Részleges

JavaScript Data types

Kérdés

Which is not a valid data type in JavaScript?

  • A Undefined
  • B Boolean
  • C Float A te válaszodhelyes
  • D Number
  • E Struct szintén helyes – kimaradt
Beadott válasz: csak Float — Hiányzik: Struct is érvénytelen! Mindkettőt be kellett volna jelölni.
Helyes válasz: Float ÉS Struct

Magyarázat

A JavaScript 8 adattípust ismer (7 primitív + 1 referencia):

// 7 primitív típus:
typeof undefined  // "undefined"
typeof true       // "boolean"
typeof 42         // "number"      ← NINCS külön Float!
typeof 42n        // "bigint"
typeof "hello"    // "string"
typeof Symbol()   // "symbol"
typeof null       // "object"      ← ismert JS bug, de null primitív

// 1 referencia típus:
typeof {}         // "object"
typeof []         // "object"      ← tömb is object
typeof function(){} // "function"  ← function is object altípus
Miért nincs Float?

A JavaScript nem különbözteti meg az egész számokat és a lebegőpontos számokat. Mindkettő Number típus (IEEE 754 double-precision 64-bit). Más nyelvekben (Java, C#) van int, float, double – JS-ben ez mind Number.

typeof 42     // "number"
typeof 3.14   // "number"  ← nincs "float"!
typeof 1e10   // "number"

// Ha egész számra van szükséged: Number.isInteger()
Number.isInteger(42)    // true
Number.isInteger(3.14)  // false
Struct sem létezik – a struct a C/C++/C#/Go nyelvek fogalma. JavaScriptben az objektum ({}) és a class a legközelebbi megfelelője.

2. JS cross-platform kompatibilitás Helyes ✓

JavaScript Platform

Kérdés

A JavaScript program developed on a Unix Machine...

  • A Will throw errors and exceptions
  • B Must be restricted to a Unix Machine
  • C Will be displayed as a JavaScript text in the browser
  • D Will work on a Windows Machine A te válaszodhelyes
Helyes válasz: D) Will work on a Windows Machine

Magyarázat

A JavaScript platform-független nyelv. A böngésző (vagy Node.js) futtatja, nem az operációs rendszer közvetlenül. Egy Unix-on írt JS kód ugyanúgy fut Windows-on, macOS-en, Linux-on – mivel mindegyiken ugyanaz a JavaScript engine fut (V8, SpiderMonkey, stb.).

Szükséges tudás: interpretált vs. fordított nyelvek, JS-motorok (V8, SpiderMonkey), a „write once, run anywhere” elv, és a futtatókörnyezet (runtime) szerepe. A böngésző absztrahálja az OS-t — a JS sosem közvetlenül a rendszerrel beszél.

3. Script attribútumok (async, defer) Helytelen ✗

HTML Script loading

Kérdés

Which attribute can affect the order of execution at page load? (script element)

  • A parse
  • B async helyes
  • C defer helyes
  • D type helyes
  • E All of the above A te válaszod
Beadott válasz: All of the above — ROSSZ, mert parse NEM létező script attribútum!
Helyes válasz: async, defer, type (NEM parse!)

Magyarázat

A <script> elem attribútumai közül a következők befolyásolják a betöltési/végrehajtási sorrendet:

NORMÁL (attribútum nélkül): HTML parsing: ████████░░░░░░░░░░░████████ Script: ↓fetch↓exec↓ Parsing BLOKKOL, amíg a script letöltődik és lefut ASYNC: HTML parsing: ████████████████░░░████████ Script: ↓fetch──↓exec↓ Letöltés párhuzamos, végrehajtás AZONNAL (blocking) Sorrend NEM garantált! DEFER: HTML parsing: ██████████████████████████ Script: ↓fetch──────↓ ↓exec↓ Letöltés párhuzamos, végrehajtás DOM PARSE UTÁN Sorrend GARANTÁLT (dokumentum sorrend) TYPE="module": HTML parsing: ██████████████████████████ Script: ↓fetch──────↓ ↓exec↓ Alapból defer viselkedés + ES module scope
<!-- Normál: blokkolja a HTML parsolást -->
<script src="app.js"></script>

<!-- Async: párhuzamos letöltés, azonnal futtat (nem garantált sorrend) -->
<script src="analytics.js" async></script>

<!-- Defer: párhuzamos letöltés, DOM parse UTÁN futtat (garantált sorrend) -->
<script src="app.js" defer></script>

<!-- Type=module: alapból defer + modul scope -->
<script type="module" src="app.mjs"></script>
parse NEM létező script attribútum! Nem szerepel a HTML specifikációban. Érvényes script attribútumok: src, type, async, defer, crossorigin, integrity, nomodule, referrerpolicy, fetchpriority.
Mikor melyiket?
  • defer — Alkalmazás fő JS fájljai (sorrend fontos, DOM kell)
  • async — Független scriptek: analytics, reklám, tracking (sorrend nem fontos)
  • type="module" — Modern ES module alapú alkalmazás
Mélyebb nüanszok (gyakori follow-up kérdés):
  • Inline scriptre NINCS hatás: az async és defer csak külső, src-vel rendelkező scriptnél érvényesül. Inline <script>...</script>-nél a böngésző figyelmen kívül hagyja őket (kivétel: type="module" inline is deferált).
  • Ha mindkettő rajta van (<script async defer>): modern böngészőben az async nyer; a defer csak fallback a régi böngészőknek, amelyek nem ismerik az async-et.
  • Több defer script: dokumentum-sorrendben, egymás után futnak, mind a DOMContentLoaded esemény ELŐTT. Több async script: amelyik előbb letöltődik, az fut előbb → a sorrend kiszámíthatatlan (verseny).
  • Az async/defer letöltése nem blokkolja a parsolást — a különbség a végrehajtás IDŐZÍTÉSÉBEN van (async: amint kész; defer: parse után, sorrendben).

4. Null check JavaScriptben Helytelen ✗

JavaScript Null check Equality

Kérdés

You have to execute a code snippet only if a is not null. How can you do it?

  • A if (!a) csapda
  • B if (a != null) csapda
  • C if (a !== null) helyes
  • D All of the above A te válaszod
Beadott válasz: All of the above — ROSSZ, mert if (!a) NEM ugyanaz, mint null check!
Helyes válasz: C) if (a !== null) — a legprecízebb "not null" ellenőrzés.

Magyarázat

let a;

// A kérdés: "futtasd a kódot, CSAK ha a NEM null"

// C) ✅ if (a !== null) — STRICT (típus is számít) "nem null" ellenőrzés
//    Kizárólag null-ra hamis, minden másra igaz → pontosan ezt kérték
a = null;      // null !== null      → false → NEM fut ✅
a = undefined; // undefined !== null → true  → FUT  ✅ (undefined ≠ null)
a = 0;         // 0 !== null         → true  → FUT  ✅
a = "";        // "" !== null        → true  → FUT  ✅
a = false;     // false !== null     → true  → FUT  ✅

// B) ⚠️ if (a != null) — LOOSE: egyszerre szűri a null-t ÉS az undefined-ot
//    (mert loose egyenlőségnél null == undefined)
a = null;      // null != null       → false → nem fut
a = undefined; // undefined != null  → false → nem fut ← itt tér el C-től!
a = 0;         // 0 != null          → true  → fut

// A) ❌ if (!a) — ez NEM null-ellenőrzés, hanem FALSY-ellenőrzés
//    Fordított logikájú: a kód akkor fut, ha a FALSY (tehát null-ra is!)
a = null;      // !null  = true  → FUT  ❌ (pont fordítva, mint kéne)
a = 0;         // !0     = true  → FUT  ❌ (0 nem null, mégis "elkapja")
a = "";        // !""    = true  → FUT  ❌ ("" nem null, mégis "elkapja")
a = 42;        // !42    = false → nem fut ❌ (42 nem null, mégis kihagyja)

Vagyis a !a a falsy értékeket fogja meg (null, undefined, 0, "", false, NaN), nem a null-t — épp az ellenkező halmaz, mint amit a kérdés kér. Még a megfordított if (a) sem lenne jó, mert az a 0-t, ""-t és false-ot is kizárná, holott azok nem null.

Érték if (!a) if (a != null) if (a !== null) Elvárt (not null)
nulltrue (fut)false (nem fut)false (nem fut)NEM fut
undefinedtrue (fut)false (nem fut)true (fut)FUT (nem null)
0true (fut)true (fut)true (fut)FUT (nem null)
""true (fut)true (fut)true (fut)FUT (nem null)
falsetrue (fut)true (fut)true (fut)FUT (nem null)
42false (nem fut)true (fut)true (fut)FUT (nem null)
if (!a) probléma: Az !a FORDÍTOTT logika — a kódot futtatja, ha a falsy (tehát null-ra is), és NEM futtatja, ha truthy. A kérdés pedig "execute if NOT null" — ez az ellentéte! Még ha megfordítanánk if (a)-ra, az sem jó: 0, "", false is kiesne, holott ezek nem null.
Modern alternatívák (ES2020+):
  • Nullish coalescing ?? — csak null/undefined esetén ad alapértéket: const val = a ?? "default"; (a 0 és "" megmarad, ellentétben a ||-lel!)
  • Optional chaining ?. — biztonságos hozzáférés: a?.prop nem dob hibát, ha a null/undefined (helyette undefined-ot ad)
  • Nullish assignment ??= — csak akkor értékel, ha a bal oldal null/undefined: a ??= defaultObj;

5. RegExp pattern test Helytelen ✗

JavaScript RegExp

Kérdés

Given:

const text = "This is an example... 1, 2, 3"; // Sample text
const pattern = /\d+/g; // Matches all instances of 1 or more digits

How would you check if the pattern can be found in text?

  • A text == pattern
  • B text.equals(pattern)
  • C text.test(pattern) A te válaszod
  • D pattern.test(text) helyes
  • E None of the above
Beadott válasz: text.test(pattern) — ROSSZ! A .test() a RegExp metódusa, nem a String-é!
Helyes válasz: D) pattern.test(text)

Magyarázat

const text = "This is an example... 1, 2, 3";
const pattern = /\d+/g;

// ✅ HELYES: RegExp.prototype.test(string)
pattern.test(text);     // true (van szám a szövegben)

// ❌ ROSSZ: String-nek NINCS .test() metódusa!
text.test(pattern);     // ❌ TypeError: text.test is not a function

// Alternatív String metódusok RegExp-hez:
text.match(pattern);    // ["1", "2", "3"]  (összes találat)
text.search(pattern);   // 24  (első találat indexe)
text.replace(pattern, 'X'); // "This is an example... X, X, X"
text.matchAll(pattern); // iterator az összes találathoz
Ökölszabály:
  • pattern.test(text) → boolean (van-e találat?)
  • text.match(pattern) → tömb (mik a találatok?)
  • text.search(pattern) → szám (hol az első találat?)
  • pattern.exec(text) → részletes találat objektum
FONTOS csapda: a /g flag állapotossá teszi a .test()-et!

A pattern itt /\d+/g (global flag). Emiatt a RegExp objektum megjegyzi a lastIndex-et a hívások között, így UGYANARRA a szövegre ismételten hívva felváltva true/false-t adhat vissza — ez gyakori, alattomos bug:

const pattern = /\d+/g;
const text = "a1b2";

pattern.test(text); // true   (talál "1"-et, lastIndex → 2)
pattern.test(text); // true   (talál "2"-t,  lastIndex → 4)
pattern.test(text); // false  (nincs több, lastIndex visszaáll 0-ra)
pattern.test(text); // true   (újra elölről kezdi!)

// Ugyanez igaz a pattern.exec(text)-re is — szintén lépteti a lastIndex-et.

Megoldások:

// 1) Ne tedd rá a /g flaget, ha CSAK létezést vizsgálsz:
/\d+/.test("a1b2");        // mindig true, állapotmentes

// 2) Vagy nullázd a lastIndex-et hívás előtt:
pattern.lastIndex = 0;
pattern.test(text);

// 3) Vagy használj állapotmentes String-metódust:
"a1b2".match(/\d+/g);      // ["1", "2"]  (g flaggel is állapotmentes)

6. Prototype chain ellenőrzés Helytelen ✗

JavaScript Prototype

Kérdés

How can you determine whether one object is the prototype of (or is part of the prototype chain of) another object?

  • A isPrototypeOf() helyes
  • B equals() method
  • C '===' operator
  • D None of the above A te válaszod
Beadott válasz: None of the above — ROSSZ! Az isPrototypeOf() pontosan erre való!
Helyes válasz: A) isPrototypeOf()

Magyarázat

// Object.prototype.isPrototypeOf() — ellenőrzi a prototype chain-t

const animal = { eats: true };
const rabbit = Object.create(animal); // rabbit prototípusa az animal

animal.isPrototypeOf(rabbit);         // ✅ true
Object.prototype.isPrototypeOf(rabbit); // ✅ true (chain végén)

// Másik módszer: instanceof (class-okhoz)
class Animal {}
class Dog extends Animal {}
const dog = new Dog();

dog instanceof Dog;    // true
dog instanceof Animal; // true (prototype chain-ben van)

// equals() NEM létezik JS-ben (az Java):
// obj1.equals(obj2) → ❌ TypeError

// === referencia-egyenlőséget vizsgál, NEM prototype chain-t:
// obj1 === obj2 → csak akkor true, ha UGYANAZ az objektum
Prototype chain vizsgálat eszközei:
  • proto.isPrototypeOf(obj) — proto benne van-e obj chain-jében?
  • obj instanceof Constructor — Constructor.prototype benne van-e obj chain-jében?
  • Object.getPrototypeOf(obj) — obj közvetlen prototípusát adja vissza

7. Object Serialization Helyes ✓

JavaScript Serialization

Kérdés

What is the process of converting an object's state to a string from which it can later be restored called?

  • A Object Encapsulation
  • B Object Serialization A te válaszodhelyes
  • C Object Abstraction
  • D Object Encoding
Helyes válasz: B) Object Serialization

Magyarázat

// Serialization: objektum → string (tárolható/küldhető formátum)
const user = { name: "Anna", age: 25 };
const json = JSON.stringify(user);   // '{"name":"Anna","age":25}'

// Deserialization: string → objektum (visszaállítás)
const restored = JSON.parse(json);   // { name: "Anna", age: 25 }

// Többi fogalom:
// Encapsulation = adatok és metódusok egy egységbe zárása (OOP)
// Abstraction   = komplexitás elrejtése egyszerű interfésszel (OOP)
// Encoding      = karakter/adat átalakítás (pl. Base64, UTF-8)
JSON.stringify korlátai (gyakori interjú-follow-up):
  • function és undefined érték kiesik objektumból (tömbben viszont null lesz a helyén)
  • Date → ISO string lesz, és a JSON.parse NEM állítja vissza Date-té (string marad)
  • Map, Set, RegExp → üres {} (nem szerializálódik az adatuk)
  • BigInt → TypeError ("Do not know how to serialize a BigInt")
  • Körkörös referencia → TypeError ("Converting circular structure to JSON")
  • NaN, Infinity → null

Egyedi szerializáláshoz: toJSON() metódus az objektumon, vagy a JSON.stringify(obj, replacer) 2. argumentuma (replacer függvény).

8. toLocaleString() Helytelen ✗

JavaScript Intl Localization

Kérdés

toLocaleString() is used to:

  • A return a localised object representation
  • B return a parsed string
  • C return a localised string representation of the object helyes
  • D return a local time in string format A te válaszod
  • E None of the above
Beadott válasz: "return a local time in string format" — Túl szűk! A toLocaleString() nem csak időhöz használható.
Helyes válasz: C) return a localised string representation of the object

Magyarázat

A toLocaleString() metódus bármilyen objektum lokalizált string reprezentációját adja vissza — nem csak dátumokét!

// SZÁM lokalizálás:
const num = 123456.789;
num.toLocaleString('hu-HU');  // "123 456,789"  (magyar formátum)
num.toLocaleString('de-DE');  // "123.456,789"  (német formátum)
num.toLocaleString('en-US');  // "123,456.789"  (amerikai formátum)

// DÁTUM lokalizálás:
const date = new Date();
date.toLocaleString('hu-HU'); // "2026. 02. 13. 10:30:00"
date.toLocaleString('en-US'); // "2/13/2026, 10:30:00 AM"

// TÖMB lokalizálás:
const arr = [1234, 5678];
arr.toLocaleString('hu-HU');  // "1 234,5 678"

// PÉNZNEM:
num.toLocaleString('hu-HU', { style: 'currency', currency: 'HUF' });
// "123 457 Ft"

num.toLocaleString('en-US', { style: 'currency', currency: 'USD' });
// "$123,456.79"
D miért rossz? "Return a local time in string format" — a toLocaleString() nem csak időre vonatkozik. Számokra, tömbre, dátumra, bármire használható. A helyes definíció: lokalizált string reprezentáció az objektumról.

9. useEffect mount-on futtatás Helyes ✓

React Hooks useEffect

Kérdés

How do you invoke setDone only when component mounts, using hooks?

function MyComponent(props) {
  const [done, setDone] = useState(false);
  return <h1>Done: {done}</h1>;
}
  • A useEffect(() => { setDone(true); });
  • B useEffect(() => { setDone(true); }, []); A te válaszodhelyes
  • C useEffect(() => { setDone(true); }, [setDone]);
  • D useEffect(() => { setDone(true); }, [done, setDone]);
Helyes válasz: B) useEffect(() => { setDone(true); }, []);

Magyarázat

// Dependency array szabályok:

useEffect(() => { ... });          // A) Minden rendereléskor fut ❌
useEffect(() => { ... }, []);      // B) Csak MOUNT-kor fut (egyszer) ✅
useEffect(() => { ... }, [setDone]); // C) Amikor setDone változik (stabil ref → egyszer)
useEffect(() => { ... }, [done, setDone]); // D) done VAGY setDone változásakor

// [] = üres dependency array = "nincs függőség" = csak mount-kor fut
// Ez ekvivalens az osztálykomponens componentDidMount()-jával
Megjegyzés: A setDone (useState setter) referenciája stabil (React garantálja), ezért C technikailag is csak egyszer futna. De a konvenció és a helyes válasz: üres dependency array [] a mount-only effect-hez.
React 18+ StrictMode — a mount-effekt DEV-ben KÉTSZER fut!

Fejlesztői módban <StrictMode> alatt a React szándékosan kétszer hívja meg a mount-effektet (mount → cleanup → újra mount), hogy kiszűrje a nem-idempotens / cleanup nélküli effekteket. Productionben csak egyszer fut.

useEffect(() => {
  const sub = api.subscribe();      // mellékhatás
  return () => sub.unsubscribe();   // ← KÖTELEZŐ cleanup, különben dupla subscription
}, []);

Tanulság: az üres [] „csak mount-kor" effektnek is mindig legyen cleanupja, ha erőforrást foglal (subscription, timer, event listener), különben StrictMode-ban (és unmountkor) szivárgás vagy dupla regisztráció lesz.

Még egy apróság ehhez a kódhoz: a <h1>Done: {done}</h1> egy boolean-t próbál renderelni — a React a true/false értéket NEM jeleníti meg (semmit sem ír ki). Ha látható szöveg kell: {done ? "igen" : "nem"} vagy {String(done)}.

10. onClick handler viselkedés Helyes ✓

React Event handling

Kérdés

What is the difference between the click behaviours of these two buttons (assuming that this.handleClick is bound correctly)?

A. <button onClick={this.handleClick}>Click Me</button>
B. <button onClick={(event) => this.handleClick(event)}>Click Me</button>
  • A Button A will not have access to the event object on click of the button
  • B Button B will not fire the handler this.handleClick successfully
  • C Button A will not fire the handler this.handleClick successfully
  • D There is no difference A te válaszodhelyes

// Button A: közvetlen referencia

// Button B: arrow function wrapper
<button onClick={(event) => this.handleClick(event)}>

  • De a tényleges kattintás viselkedés azonos.

11. React.PureComponent Helytelen ✗

React PureComponent Performance

Kérdés

When might you use React.PureComponent?

  • A When you do not want your component to have state A te válaszod
  • B When you have sibling components that need to be compared
  • C When you want a default implementation of shouldComponentUpdate() helyes
  • D When you do not want your component to have props
Beadott válasz: "do not want your component to have state" — ROSSZ! A PureComponent-nek LEHET state-je.
Helyes válasz: C) When you want a default implementation of shouldComponentUpdate()

Magyarázat

// React.Component — NEM optimalizált (mindig renderel)
class MyComponent extends React.Component {
  // shouldComponentUpdate() → alapból MINDIG true
  // → minden parent rendereléskor újrarenderel
}

// React.PureComponent — OPTIMALIZÁLT (shallow comparison)
class MyComponent extends React.PureComponent {
  // shouldComponentUpdate() → AUTOMATIKUS shallow compare
  // → csak akkor renderel, ha props VAGY state VÁLTOZOTT
  // (shallow === összehasonlítás minden prop/state mezőn)
}

// Példa: mikor NEM renderel újra a PureComponent?
<MyPure name="Anna" age={25} />
// Ha a parent újrarenderel, de name="Anna" és age=25 nem változott
// → PureComponent KIHAGYJA az újrarenderelést ✅

// Modern ekvivalens (function component):
const MyComponent = React.memo(function MyComponent(props) {
  // Csak akkor renderel, ha a props megváltozott (shallow compare)
  return <div>{props.name}</div>;
});
Gyakori tévhit: "PureComponent = stateless". HAMIS! A PureComponent-nek teljes értékű state-je lehet. A "pure" itt azt jelenti: shallow comparison alapú renderelés optimalizálás, nem "állapot nélküli". Ha state-mentes komponenst akarsz, használj egyszerű function component-et.
A shallow compare buktatója: a mutáció „láthatatlan"!

A PureComponent/React.memo csak felületes (referencia szintű) összehasonlítást végez. Ha egy objektum/tömb propot vagy state-et mutálsz (a referencia ugyanaz marad), a komponens NEM renderel újra, pedig a tartalom megváltozott:

// ❌ Mutáció — a referencia ugyanaz, PureComponent nem veszi észre:
this.state.items.push(newItem);
this.setState({ items: this.state.items });  // ugyanaz a tömb-ref → NINCS re-render

// ✅ Új referencia — a shallow compare észleli a változást:
this.setState({ items: [...this.state.items, newItem] });

Ezért a PureComponent/memo használata immutable adatkezelést feltételez. Mély (nested) objektumoknál a shallow compare szintén félrevezethet — ott vagy lapítsd az adatot, vagy adj egyedi összehasonlítót a React.memo 2. argumentumában (areEqual).

12. React keys egyedisége Helyes ✓

React Keys Lists

Kérdés

Keys are given to a list of elements in React. These keys should be -

  • A Unique in the DOM
  • B Do not require to be unique
  • C Unique among the siblings only A te válaszodhelyes
  • D All of the above
Helyes válasz: C) Unique among the siblings only

Magyarázat

// ✅ HELYES: Key-k csak a TESTVÉREK között kell egyediek legyenek
function App() {
  return (
    <div>
      {/* Lista 1 */}
      <ul>
        <li key="1">Alma</li>    {/* key="1" itt OK */}
        <li key="2">Körte</li>
      </ul>

      {/* Lista 2 — UGYANAZ a key más helyen: OK! */}
      <ul>
        <li key="1">Piros</li>   {/* key="1" itt is OK, más szülő */}
        <li key="2">Kék</li>
      </ul>
    </div>

}

// ❌ ROSSZ: azonos key testvérek között
<ul>
  <li key="1">Alma</li>
  <li key="1">Körte</li>   {/* ❌ Duplikált key a testvérek között! */}

// ⚠️ Index-et kerüld key-ként (ha a lista sorrendje változhat):
items.map((item, index) => <li key={index}>...</li>);  // ⚠️ fragile
items.map(item => <li key={item.id}>...</li>);          // ✅ stabil
Szükséges tudás: reconciliation (egyeztetési algoritmus), virtuális DOM, miért rossz ötlet a tömb-index kulcsként (sorrend-változásnál hibás újrahasznosítás, state-keveredés), és stabil, egyedi azonosítók használata.

13. Controlled input element Helyes ✓

React Forms Controlled component

Kérdés

What is a controlled input element?

  • A An input element with the controlled flag
  • B An input element whose value is being controlled by some state A te válaszodhelyes
  • C An input element that can only accept a list of characters
  • D An input element that is controlled by the value of another input element
Helyes válasz: B) An input element whose value is being controlled by some state

Magyarázat

// ✅ CONTROLLED input — React state az "igazság forrása"
function ControlledForm() {
  const [name, setName] = useState("");

  return (
    <input
      value={name}                          // state-ből jön az érték
      onChange={(e) => setName(e.target.value)} // state frissül inputra
    />
  );
  // A React state KONTROLLÁLJA az input értékét
  // Az input mindig a state-et tükrözi
}

// ❌ UNCONTROLLED input — a DOM az "igazság forrása"
function UncontrolledForm() {
  const inputRef = useRef(null);

    console.log(inputRef.current.value); // DOM-ból olvassuk
  };

  return <input ref={inputRef} />;
  // A DOM kezeli az értéket, nem a React state
}
