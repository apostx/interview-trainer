Futási idő & memória – a Big O (ordó) érthetően

Mit jelent O(1), O(log n), O(n), O(n²)… és miért számít? Ez a dokumentum a komplexitás-elemzést építi fel a nulláról: egyszerű, gyorsan érthető példákkal, sok ábrával, és egy kiemelt esettanulmánnyal (Palindrome). A cél: ránézésre meg tudd mondani egy kód futási idejét és memóriaigényét.

1. Miért fontos? – a skálázódás

Bevezetés

Egy algoritmus jó lehet 10 elemnél, de összeomolhat 10 milliónál. A komplexitás-elemzés arra ad választ: hogyan nő a futási idő (és a memória), ha a bemenet (n) nő? Nem azt mérjük, hogy „hány másodperc" (az gép-, nyelv- és terhelésfüggő), hanem azt, hogy hányszorosára nő a munka, ha a bemenet duplázódik.

Hasonlat: ha 1 telefonkönyv-oldalt 1 mp alatt nézel át, akkor 1000 oldal lineárisan 1000 mp. De ha „felezős" kereséssel (binary search) keresel, 1000 oldal is csak ~10 lépés. Ugyanaz a feladat, drámaian más skálázódás — ezt méri a Big O.

Az interjún tipikus kérdés: „mi ennek a függvénynek az időkomplexitása?" — a válaszhoz meg kell tanulni „leolvasni" a kódból a növekedési osztályt.

2. Mi az a Big O? – az alapszabályok

Definíció

A Big O (ordó) egy növekedési ütem felső korlátja nagy n-re (általában a legrosszabb esetre). Nagyon fontos: ez egy jelölés, amit kétféle erőforrásra alkalmazunk — a futási időre (CPU-műveletek száma) és a memóriára (tér). Egy algoritmusnak tehát mindig két, egymástól független komplexitása van, és a kettőt külön adjuk meg (pl. „O(n) idő, O(1) tér"). A lenti osztályok és szabályok mindkettőre érvényesek. Két egyszerű szabály alapján egyszerűsítünk:

  • 1. Elhagyjuk a konstans szorzókat. 5n → O(n), 3n² → O(n²). (Nem érdekel, hogy 5× vagy 1× — a növekedés alakja számít.)
  • 2. Csak a domináns tag marad. n² + 100n + 500 → O(n²), mert nagy n-nél a n² minden mást „lehagy".

n=5-nél a tagok még összemérhetők; n=50-nél a négyzetes tag már sokszorosa a többinek. Ezért elég a domináns tagot megadni: O(n²).

Gyakori félreértés: a Big O nem a pontos futási idő, és nem jelenti, hogy „mindig lassú". Egy O(n²) kód kis n-re gyorsabb lehet, mint egy O(n) sok konstans overhead-del. A Big O a skálázódásról szól nagy n-re.

3. A komplexitási osztályok – a nagy grafikon

Áttekintés

Ez a leghasznosabb egyetlen ábra: hogyan nő a műveletek száma a bemenet méretével az egyes osztályoknál. Minél „laposabb" a görbe, annál jobban skálázódik.

A „jó" osztályok (O(1), O(log n)) szinte laposak — alig nő a munka. Az O(n²) és különösen az O(2ⁿ) robban: kis n-re még jó, de gyorsan használhatatlanná válik.

Gyorsasági sorrend (a leggyorsabbtól a leglassabbig): O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!).

4. O(1) – konstans idő

Kiváló

A futási idő nem függ a bemenet méretétől. Akár 10, akár 10 millió elem van, ugyanannyi lépés. Tipikus: tömb-indexelés, hash-map lekérdezés, egy aritmetikai művelet.

A tömb a memóriában folytonos: a 3. elem címe kiszámolható (alapcím + 3×elemméret), így egyetlen lépésben elérhető — függetlenül a tömb hosszától.

function first(arr) { return arr[0]; }       // O(1)
const value = map.get(key);                  // O(1) átlagosan (hash)
const isEven = (n) => n % 2 === 0;            // O(1)

5. O(log n) – logaritmikus (felezés)

Kiváló

Minden lépésben megfelezzük a problémát. Ezért a lépések száma a log₂(n) nagyságrendje: 1000 elem → ~10 lépés, 1 000 000 → ~20 lépés. Klasszikus példa: bináris keresés rendezett tömbben.

16 elemet 4 felezés talál meg (log₂16 = 4). Minden lépés a maradék felét dobja el — ezért nő a lépésszám csak logaritmikusan.

function binarySearch(sorted, target) {       // O(log n)
  let lo = 0, hi = sorted.length - 1;
  while (lo <= hi) {
    const mid = Math.floor((lo + hi) / 2);
    if (sorted[mid] === target) return mid;
    if (sorted[mid] < target) lo = mid + 1;   // a felső felében keress
    else hi = mid - 1;                        // az alsó felében keress
  }
  return -1;
}
Ismerd fel: ha minden lépésben felezel/harmadolsz (nem egyesével haladsz), az log. Kiegyensúlyozott fa (BST) keresése, beszúrása szintén O(log n).

6. O(n) – lineáris  ·  esettanulmány: Palindrome

Jó Palindrome

A munka egyenes arányban nő a bemenettel: ha duplázod n-t, duplázódik a munka. Tipikus: egyetlen ciklus végig az adatokon (összeg, maximum, keresés rendezetlenben).

function sum(arr) {            // O(n) – minden elemet egyszer érint
  let total = 0;
  for (const x of arr) total += x;
  return total;
}

Esettanulmány: Palindrome-ellenőrzés (két mutató)

Palindróm = előről és hátulról ugyanaz (pl. "racecar", "géppé"). A leghatékonyabb megoldás a két-mutató (two-pointer) technika: egy mutató elölről, egy hátulról közelít a közép felé, és párba állítva összehasonlít.

A két mutató kívülről befelé halad; minden lépésben egy párt hasonlít össze. Összesen ~n/2 összehasonlítás → O(n) idő. Extra adatszerkezet nincs → O(1) memória.

function isPalindrome(s) {        // ⏱ O(n) idő, 💾 O(1) memória
  let left = 0, right = s.length - 1;
  while (left < right) {
    if (s[left] !== s[right]) return false;  // pár nem egyezik → nem palindróm
    left++;                       // balról jobbra
    right--;                      // jobbról balra
  }
  return true;                    // a mutatók találkoztak → palindróm
}

isPalindrome("racecar"); // true
isPalindrome("hello");   // false
Összevetés – ugyanaz az idő, más memória: 
// "Naiv" megoldás: megfordítjuk a stringet
function isPalindromeNaive(s) {   // ⏱ O(n) idő, de 💾 O(n) MEMÓRIA!
  return s === s.split('').reverse().join('');
  //          ↑ új tömb + új string jön létre (n méretű másolat)
}
Mindkettő O(n) időben, de a naiv verzió O(n) extra memóriát használ (másolat), míg a két-mutató O(1)-et. Interjún ezt a különbséget érdemes kimondani — a komplexitásnak két dimenziója van: idő ÉS tér.

7. O(n log n) – linearitmikus

Jó (rendezés)

A hatékony összehasonlító rendezések (merge sort, quicksort átlagosan, heapsort) ide esnek. Intuíció: log n szintnyi felezés, és minden szinten n munka (összefésülés) → n × log n.

A felosztás log n szintet hoz létre (mindig felezünk), és minden szinten összesen n elemet fésülünk össze → O(n log n). Ez a gyakorlatban a leggyorsabb általános rendezés nagyságrendje.

arr.sort((a, b) => a - b);   // a JS motor O(n log n) rendezést használ

8. O(n²) – négyzetes (egymásba ágyazott ciklus)

Vigyázz nagy n-re

Tipikus jel: ciklus a cikluson belül, mindkettő n-ig fut. Minden elemet minden elemmel összevetünk → n × n művelet. Példák: bubble sort, naiv duplikátum-keresés, minden pár vizsgálata.

Két egymásba ágyazott n-es ciklus n² műveletet ad. n=6 → 36 pont; n=1000 → 1 000 000 művelet. Ezért lassul be drámaian nagy bemenetnél.

function hasDuplicate(arr) {       // O(n²) – minden párt megvizsgál
  for (let i = 0; i < arr.length; i++) {
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) return true;
    }
  }
  return false;
}

// ✅ Jobb: Set-tel O(n) idő (O(n) memória árán):
function hasDuplicateFast(arr) {   // O(n)
  const seen = new Set();
  for (const x of arr) { if (seen.has(x)) return true; seen.add(x); }
  return false;
}
Tipikus optimalizálás: O(n²) → O(n) gyakran úgy, hogy memóriát áldozunk időért (hash-halmaz / map). Ez a klasszikus „space–time trade-off".

9. O(2ⁿ) és O(n!) – exponenciális / faktoriális

Kerülendő

Minden lépésnél „elágazik" a munka → robban. O(2ⁿ): minden elemre két ág (pl. naiv rekurzív Fibonacci, részhalmazok). O(n!): minden permutáció (pl. utazó ügynök brute-force). Már n=40-nél is csillagászati.

A naiv fib(n) minden hívásnál kétfelé ágazik, és ugyanazokat a részeredményeket sokszor újraszámolja → O(2ⁿ). A fa mérete a magassággal duplázódik.

function fib(n) {                 // ❌ O(2ⁿ) – exponenciális
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2);
}

// ✅ Memoizációval O(n) idő (a részeredményeket eltároljuk):
function fibMemo(n, memo = {}) {
  if (n <= 1) return n;
  if (memo[n]) return memo[n];
  return memo[n] = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
}

O(n!) – faktoriális (minden permutáció)

n elem összes sorrendje (permutációja) n! darab, ezért a generálásuk legalább O(n!). Még gyorsabban robban, mint a 2ⁿ:

function permutations(arr) {        // ❌ O(n!) – n elem összes sorrendje
  if (arr.length <= 1) return [arr];
  const result = [];
  for (let i = 0; i < arr.length; i++) {
    const rest = [...arr.slice(0, i), ...arr.slice(i + 1)];
    for (const p of permutations(rest)) result.push([arr[i], ...p]);
  }
  return result;
}
permutations([1, 2, 3]); // 3! = 6 sorrend
// kimenet mérete: 3!=6, 4!=24, 10!=3 628 800 … 20! ≈ 2,4×10¹⁸ — kezelhetetlen
// Tipikus: utazó ügynök (TSP) brute-force = minden útvonalat kipróbál
Mennyire rossz? O(2ⁿ): n=50 → ~10¹⁵ művelet (évek). O(n!): n=20 → ~2,4×10¹⁸. Ha exponenciálist látsz, szinte mindig van jobb megoldás (memoizáció, dinamikus programozás, mohó/greedy).

10. Tér-komplexitás (memóriaigény)

Memória

A tér-komplexitás azt méri, mennyi extra memória kell a bemeneten felül, ahogy n nő. Fő források: új adatszerkezetek (tömb, map, halmaz) és a rekurziós hívási verem (call stack).

Ugyanaz a jelölés, mint az időnél! A memóriaigényt pontosan ugyanazzal az ordó-jelöléssel írjuk le, mint a CPU-időt — beleértve a nagy és kis ordót is. Tehát van értelme azt mondani, hogy egy algoritmus O(1) / O(n) / O(n²) memóriát használ, vagy hogy az extra memóriája o(n) (szigorúan szublineáris). A nagy↔kis ordó különbség (megengedő ≤ vs. szigorú <) ugyanúgy érvényes a memóriára, mint az időre.

Idő- és tér-komplexitás külön (ugyanarra az algoritmusra)

AlgoritmusIdő (CPU)Tér (memória)
Palindrome – két mutatóO(n)O(1) (csak 2 index)
Palindrome – „megfordítós"O(n)O(n) (másolat készül)
Bináris keresés (iteratív)O(log n)O(1)
Merge sortO(n log n)O(n) (segéd-tömb)
Rekurzív faktoriálisO(n)O(n) (hívási verem)

Figyeld meg: az idő és a tér nem ugyanaz — pl. a két-mutatós Palindrome és a „megfordítós" időben azonos (O(n)), de memóriában nem (O(1) vs O(n)).

  • O(1) tér: néhány változó, a mérettől függetlenül (pl. a Palindrome két mutatója, egy összegző ciklus).
  • O(n) tér: a bemenettel arányos másolat/struktúra (pl. arr.map(...), egy Set az összes elemmel, a string megfordítása).
  • O(rekurzió mélysége) tér: minden rekurzív hívás egy keretet tesz a stackre — a mélység a memória.

A rekurzió a hívási vermet használja: factorial(n) n keretet rak egymásra → O(n) tér-komplexitás (még ha O(1) memóriát „látszik" is használni). Túl mély rekurzió → „stack overflow".

Idő ≠ tér: egy algoritmusnak külön van idő- és tér-komplexitása. A Palindrome két-mutató: O(n) idő, O(1) tér. A naiv „megfordítós": O(n) idő, O(n) tér. Interjún mindig add meg mindkettőt.

11. Hogyan számold ki? – gyakorlati szabályok

Gyakorlat
  • Egymás utáni lépések → összeadás (a domináns marad): O(n) + O(n) = O(n); O(n) + O(n²) = O(n²).
  • Egymásba ágyazott ciklusok → szorzás: külső n × belső n = O(n²); külső n × belső log n = O(n log n).
  • Konstans szorzók elhagyhatók: O(2n) = O(n); a fél tömbön végigmenni (n/2) is O(n).
  • Felezés/harmadolás ciklusban → log; rekurzió, ami felezi a bemenetet → log szint.
  • Külön bemenetek: ha két különböző méret van (a és b), ne mosd össze: O(a + b) vagy O(a × b), ne O(n²).
function example(arr) {           // elemzés:
  console.log(arr[0]);            // O(1)
  for (const x of arr) sum += x;  // O(n)
  for (let i = 0; i < arr.length; i++)
    for (let j = 0; j < arr.length; j++)
      pair(arr[i], arr[j]);       // O(n²)
}
// Összesen: O(1) + O(n) + O(n²) = O(n²)  (a domináns tag nyer)

És a „két különböző bemenet" szabály konkrétan — ne keverd egy n-be a két méretet (a és b):

function compare(listA, listB) {     // ✅ O(a + b) — NEM O(n²)!
  for (const x of listA) check(x);   // O(a)
  for (const y of listB) check(y);   // O(b)   → egymás után: a + b
}

function pairs(listA, listB) {       // O(a × b) — minden A–B pár
  for (const x of listA)
    for (const y of listB) combine(x, y);   // egymásba ágyazva: a × b
}

Big O vs Θ (Theta) vs Ω (Omega) – best/avg/worst

JelölésMit mondPélda (rendezetlen keresés)
O (Big O)felső korlát – „legrosszabb eset"O(n) — az elem a végén / nincs benne
Ω (Omega)alsó korlát – „legjobb eset"Ω(1) — az elem rögtön az elején van

A gyakorlatban szinte mindig a Big O-t (legrosszabb eset) használjuk, mert garanciát ad. „Átlagos eset" akkor jön elő, amikor a worst case félrevezető (pl. quicksort: átlag O(n log n), worst O(n²)).

„Kis" jelölések: kis ordó (little-o) és kis omega (ω) – szigorú korlátok

A fenti három a „nagy" jelölés. Mindegyiknek van „kis" párja is, amelyek szigorú (nem szoros) korlátot jelentenek. A legkönnyebb a számoknál ismert relációkkal párba állítani őket:

JelölésReláció-analógiaJelentésPélda
O (nagy O)≤felső korlát — lehet szoros isn = O(n),   n = O(n²)
o (kis ordó)< (szigorú)szigorúan lassabban nő — a korlát NEM szorosn = o(n²) ✓,   de n ≠ o(n) ✗
Ω (nagy omega)≥alsó korlát — lehet szoros isn = Ω(n),   n = Ω(log n)
ω (kis omega)> (szigorú)szigorúan gyorsabban nőn² = ω(n) ✓,   de n ≠ ω(n) ✗
Θ (theta)=pontosan azonos nagyságrend (O és Ω egyszerre)n = Θ(n)

A kulcskülönbség O és o (nagy és kis ordó) között: a nagy O megengedi az egyenlőséget (n = O(n) igaz), a kis ordó nem (n ≠ o(n), mert n nem nő szigorúan lassabban önmagánál). Intuitíven: f(n) = o(g(n)) azt jelenti, hogy f(n) / g(n) → 0, ahogy n → ∞ — vagyis a g idővel teljesen „elhúz" f mellől.

Kis ordó (f = o(g)): ahogy n nő, a g(n) egyre jobban elhúz f(n) fölött, az arányuk 0-hoz tart. (Nagy O-nál a két görbe akár együtt is futhat — ezért igaz pl. n = O(n), de nem igaz n = o(n).)

Mikor kell? A gyakorlati interjún ritkán — ott a nagy O a sláger. A kis ordó / kis omega inkább elméleti (algoritmus-elemzés, tételek, pl. „ez a tag o(n), tehát elhanyagolható nagy n-re"). De jó pont, ha tudod a lényeget: a „nagy" jelölések megengedők (≤ / ≥), a „kis" jelölések szigorúak (< / >).

13. Interjúkérdések

Interjú
K1: Mi a Palindrome-ellenőrzés idő- és tér-komplexitása?

Két-mutató megoldással O(n) idő (minden karaktert legfeljebb egyszer érintünk, ~n/2 összehasonlítás) és O(1) tér (csak két index). A „megfordítós" megoldás szintén O(n) idő, de O(n) extra tér a másolat miatt.

K2: Miért O(log n) a bináris keresés?

Mert minden lépésben a keresési tér felére csökken. log₂(n) felezés után marad 1 elem, tehát ~log₂(n) lépés. Feltétel: a tömb rendezett.

K3: Hogyan tennél egy O(n²)-es duplikátum-keresést gyorsabbá?

Set-tel: végigmegyünk egyszer, és minden elemet egy halmazba teszünk; ha már benne van → duplikátum. Így O(n) idő, O(n) tér — időt nyerünk memória árán (space–time trade-off).

K4: Mi a különbség O(n) és O(1) tér között egy ciklusnál?

Ha a ciklus csak néhány változót használ (összeg, számláló) → O(1) tér. Ha új, n-nel arányos struktúrát épít (másolat-tömb, map) → O(n) tér. A futási idő mindkettőnél lehet O(n).

Interjú tipp: mindig (1) mondd ki az idő- ÉS a tér-komplexitást is, (2) említsd a legrosszabb esetet, (3) ha tudsz, ajánlj egy jobb komplexitású megoldást és a trade-offot. Az „elhagyom a konstansokat és a kisebb tagokat" indoklás mindig jó pont.
