1. kérdés – == (loose equality) típuskoerció

JavaScript Type coercion

Kérdés

Choose the correct answer with the result of C in JavaScript:

A = 10;
B = '10';
C = (A == B);
  • A false
  • B true
  • C 1
Helyes válasz: B) true

Magyarázat

A == operátor (loose equality / abstract equality) típuskonverziót végez az összehasonlítás előtt. Mivel az A egy szám (10) és a B egy string ('10'), a JavaScript a string-et számmá konvertálja: Number('10') → 10. Ezután 10 == 10 → true.

// A háttérben ez történik:
10 == '10'
10 == Number('10')
10 == 10
// → true

// Ha === (strict equality) lenne:
10 === '10'  // → false (nincs típuskonverzió, szám !== string)
Becsapós opciók:
  • A) false – Erre gondolnál, ha összekevered a ==-t a ===-vel. A === (strict equality) tényleg false-t adna, mert nem végez típuskonverziót. De a kérdésben == van!
  • C) 1 – Egyes nyelvekben (pl. C, PHP régi verziók) a boolean összehasonlítás eredménye 1 vagy 0 szám. A JavaScriptben az == eredménye mindig boolean (true/false), nem szám. Bár Number(true) === 1, maga az eredmény true, nem 1.
Ökölszabály: Mindig használj ===-t (strict equality), hacsak nincs nagyon jó okod a ==-re. Az ESLint eqeqeq szabály pont ezt kényszeríti ki.

2. kérdés – Melyik NEM valós string metódus?

JavaScript String methods

Kérdés

Which of the following is not a valid string method in JavaScript?

  • A .padStart()
  • B .padLeft()
  • C .trim()
  • D .substr()
  • E .localeCompare()
  • F .includes()
  • G .str()
Helyes válasz: G) .str() – és valószínűleg B) .padLeft() is helyes válasz (mindkettő nem létezik).

Magyarázat

A .str() egyszerűen nem létezik a JavaScript String prototípusán. Ez a legnyilvánvalóbban hamis opció. A .padLeft() szintén nem létezik – a helyes metódus neve .padStart(). Ha a kérdés csak egy választ enged, a .str() a legbiztosabb, mert az teljesen kitalált név. A .padLeft() egy gyakori tévedés – más nyelvekben (pl. Python, C#) létezik ilyen, de JS-ben nem.

// Valós string metódusok:
"hello".padStart(10, ".")    // ".....hello"  – ES2017 (ES8)
"hello".padEnd(10, ".")      // "hello....."
"  hello  ".trim()           // "hello"
"hello".substr(1, 3)         // "ell" (⚠️ deprecated – .substring() ajánlott)
"a".localeCompare("b")       // -1 (nyelvi összehasonlítás)
"hello".includes("ell")      // true – ES2015 (ES6)

// NEM létezők:
"hello".padLeft(10)          // ❌ TypeError – nincs .padLeft()
"hello".str()                // ❌ TypeError – nincs .str()
Becsapós opciók:
  • B) .padLeft() – Ez a legrafináltabb csapda! Más nyelvekben létezik (PadLeft C#-ban), és logikus lenne a neve, de JS-ben .padStart() / .padEnd() a helyes (nem left/right). Ha rohanásban vagy, könnyen valósnak nézheted.
  • D) .substr() – Sokan gondolják, hogy ez már nem valid, mert deprecated (elavult). De deprecated ≠ nem létezik! A .substr() még mindig működik minden böngészőben, csak nem ajánlott. A helyes alternatíva: .substring() vagy .slice().

3. kérdés – String + Number összeadás

JavaScript Type coercion

Kérdés

Please choose the correct answer with the result of the following calculation:

var a = "10";
var b = 0.11;
// What is the output of a+b?
  • A 10.11
  • B 100.11
  • C Error
Helyes válasz: B) 100.11 — pontosabban a "100.11" string. A + operátor string operandus mellett összefűz (konkatenál), nem összead: "10" + 0.11 → "10" + "0.11" → "100.11".

Magyarázat – pontosan mi történik?

Az a egy string ("10"), a b egy szám (0.11). Amikor a + operátor egyik oldalán string van, a JavaScript string konkatenációt végez (nem numerikus összeadást). Tehát a b számot string-gé konvertálja, és összefűzi:

var a = "10";   // string
var b = 0.11;   // number

a + b
// 1. a + operátor: az egyik oldal string → konkatenáció
// 2. b-t string-gé konvertálja: String(0.11) → "0.11"
// 3. "10" + "0.11" → "100.11"

console.log(a + b);        // "100.11" (string!)
console.log(typeof (a + b)); // "string"

// Ha NUMERIKUS összeadást akarnál:
console.log(Number(a) + b);  // 10.11 (number)
console.log(+a + b);         // 10.11 (unary + konvertál számmá)
console.log(parseInt(a) + b); // 10.11
Becsapós opciók:
  • A) 10.11 – Erre gondolnál, ha nem veszed észre, hogy az a string, és "normális" matematikai összeadást várnál. De a + operátor string mellett MINDIG konkatenál. Az eredmény "100.11", nem 10.11. (Megjegyzés: ha a kérdés készítői "100.11"-et akartak írni, akkor ez a helyes; ha szám 10.11-et, akkor a kérdés félrevezető.)
  • B) 100.11 – Ez valójában a helyes kimenet szövegesen! A "10" + "0.11" = "100.11". Ha a válaszok string-ként értendők, ez a jó válasz.
  • C) Error – JavaScript soha nem dob hibát string + szám összeadásnál. Más nyelvek (pl. Java strict, Python) hibát dobnának, de a JS csendben konvertál. Ez a "biztos rossz" opció.
Fontos: A kérdés válaszlehetőségei alapján a B) 100.11 a tényleges helyes kimenet (string konkatenáció: "10" + "0.11" = "100.11"). A teszt valószínűleg erre teszteli, hogy észreveszed-e: a + string mellett nem összeadás, hanem összefűzés.

4. kérdés – Template literal szintaxis

ES6 Template literals

Kérdés

Which expression evaluates to Hello, ES6!, if x = "ES6"?

  • A `Hello, $x!`
  • B '''Hello, ${x}!'''
  • C '''Hello, $x!'''
  • D `Hello, ${x}!`
  • E '''Hello, $(x)!'''
  • F `Hello, $(x)!`
Helyes válasz: D) `Hello, ${x}!`

Magyarázat

Az ES6 template literal szintaxisához két dolog kell egyszerre:

  1. Backtick (`) – a string határolója (NEM aposztróf ' és nem idézőjel ")
  2. ${kifejezés} – a változó/kifejezés beágyazása dollárjellel és kapcsos zárójelben
const x = "ES6";

// ✅ HELYES:
`Hello, ${x}!`          // "Hello, ES6!"

// ❌ ROSSZ – hiányzik a kapcsos zárójel:
`Hello, $x!`            // "Hello, $x!"  (szó szerint "$x"-et ír ki)

// ❌ ROSSZ – aposztróf nem template literal:
'Hello, ${x}!'          // "Hello, ${x}!" (szó szerint, nincs interpoláció)
'''Hello, ${x}!'''      // SyntaxError (JS-ben nincs triple-quote!)

// ❌ ROSSZ – kerek zárójel nem jó:
`Hello, $(x)!`          // "Hello, $(x)!" (szó szerint)
Becsapós opciók:
  • A) `Hello, $x!` – A legveszélyesebb csapda! Backtick-et használ (jó), és dollárjelet (jó), de hiányzik a {}. A $x szó szerint jelenik meg, nem a változó értéke. Rohanásban könnyen kihagyod a kapcsos zárójelet.
  • B, C, E) '''...''' – Pythonból ismert triple-quote szintaxis. JavaScriptben nem létezik! Három aposztróf egymás után SyntaxError.
  • F) `Hello, $(x)!` – Backtick-et használ (jó), de kerek zárójelet a kapcsos helyett. Bash/shell scriptben használatos a $() szintaxis, de JS-ben ${} kell.

5. kérdés – Melyik NEM valós JavaScript operátor?

JavaScript Operátorok

Kérdés

Which of the following is NOT a valid JavaScript operator?

  • A &&
  • B ||
  • C **
  • D !!
  • E ??
  • F ^^
Helyes válasz: F) ^^ – ez NEM létezik JavaScriptben.

Magyarázat

Nézzük végig az összes opciót:

// A) && – logikai ÉS (logical AND) ✅ LÉTEZIK
true && false   // false
"hello" && 42   // 42 (short-circuit: ha az első truthy, a másodikat adja vissza)

// B) || – logikai VAGY (logical OR) ✅ LÉTEZIK
false || "fallback"  // "fallback"
null || "default"    // "default"

// C) ** – hatványozás (exponentiation) ✅ LÉTEZIK (ES2016 / ES7)
2 ** 10    // 1024
3 ** 2     // 9
// Egyenértékű: Math.pow(2, 10)

// D) !! – dupla negáció (boolean-ra konvertál) ✅ LÉTEZIK
// Technikusan két ! operátor egymás után, nem egy önálló operátor
!!"hello"  // true
!!0        // false
!!null     // false
!!""       // false

// E) ?? – nullish coalescing ✅ LÉTEZIK (ES2020)
null ?? "default"      // "default"
undefined ?? "default" // "default"
0 ?? "default"         // 0 (⚠️ a 0 NEM nullish!)
"" ?? "default"        // "" (⚠️ az üres string NEM nullish!)
// Különbség a ||-tól: a ?? CSAK null/undefined-ra reagál

// F) ^^ – NEM LÉTEZIK ❌
// A ^ (egyetlen kalap) a BITWISE XOR operátor:
5 ^ 3      // 6  (0101 XOR 0011 = 0110)
// De a ^^ (dupla kalap) NEM operátor!
// Logikai XOR-t kézzel kell: (a || b) && !(a && b)
// Vagy: Boolean(a) !== Boolean(b)
Becsapós opciók:
  • C) ** – Sokan nem ismerik, mert viszonylag új (ES2016). Hajlamosak azt gondolni, hogy csak Math.pow() létezik. De a ** teljesen valós operátor.
  • D) !! – Trükkös, mert technikusan nem egy operátor, hanem a ! operátor kétszeri alkalmazása. Mégis, a !! mint kifejezés tökéletesen valid JS szintaxis, és széles körben használt boolean konverzióra.
  • E) ?? – Viszonylag új (ES2020), sokan nem ismerik. Könnyen gondolhatod, hogy kitalált, de létező és nagyon hasznos operátor (nullish coalescing).
  • F) ^^ – A csapda az, hogy a && és || mintájára logikus lenne egy ^^ (logikai XOR). De a JS-ben nincs logikai XOR operátor – csak a bitwise ^ létezik.

6. kérdés – const tömb módosítása

ES6 const Referencia vs érték

Kérdés

Which expression(s) does NOT throw an error, if const foo = [123, 456] in JavaScript?

  • A foo.push(789)
  • B foo[1] = 789
  • C foo = 789
  • D let bar = foo
Helyes válasz: A), B) és D) – ezek NEM dobnak hibát.
Csak C) dob hibát (TypeError: Assignment to constant variable).

Magyarázat

A const nem azt jelenti, hogy az érték "fagyasztott" (immutable) – csak azt, hogy a változó referenciája nem írható felül. A tömb (és objektum) belső tartalma szabadon módosítható, mert a referencia (a memóriacím) nem változik – csak az, amire mutat.

const foo = [123, 456];

// A) foo.push(789) ✅ NEM dob hibát
foo.push(789);
console.log(foo); // [123, 456, 789]
// A tömb TARTALMA változott, de a referencia (memóriacím) ugyanaz maradt

// B) foo[1] = 789 ✅ NEM dob hibát
foo[1] = 789;
console.log(foo); // [123, 789]
// Egy meglévő elem felülírása – a referencia nem változik

// C) foo = 789 ❌ TypeError!
foo = 789;
// TypeError: Assignment to constant variable
// A const referenciáját próbáljuk felülírni → TILOS

// Ezek is hibát dobnának:
foo = [1, 2, 3];   // ❌ – új tömb referencia
foo = [];           // ❌ – új üres tömb referencia

// D) let bar = foo ✅ NEM dob hibát
let bar = foo;
// Ez nem módosítja a foo-t! Egy ÚJ változó (bar) kap referenciát
// ugyanarra a tömbre. A const foo érintetlen marad.
console.log(bar === foo); // true – ugyanarra a tömbre mutat!
Becsapós opciók:
  • A) foo.push(789) – Sokan gondolják, hogy const = "semmit sem lehet módosítani". De a const csak a referenciát (binding-ot) védi, nem a tartalmat. A .push() a tömb tartalmát módosítja, nem a referenciát → működik.
  • B) foo[1] = 789 – Ugyanaz a félreértés: a tömbelem felülírása a tartalom módosítása, nem a referenciáé → működik.
  • D) let bar = foo – Ez nem is módosítja a foo-t semmilyen módon – egyszerűen egy új változó kap értéket. Nincs köze a const korlátozáshoz.
Ha valóban immutable tömböt akarsz: 
const foo = Object.freeze([123, 456]);
foo.push(789);   // ❌ TypeError (strict mode-ban)
foo[0] = 999;    // ❌ TypeError (strict mode-ban)
// De ez is shallow freeze – beágyazott objektumokra nem hat!

7. kérdés – Array method chaining (from, reverse, fill, map, filter, reduce)

JavaScript ES6 Array methods

Kérdés

What should be put in the places of __A__, __B__, and __C__ to obtain 42 as a result of the following computation in JavaScript?

Array
  .from({__A__: 6}, (value, index) => index)
  .reverse()
  .fill(1, 2, 3)
  .map(value => value - 1)
  .filter(value => __B__)
  .reduce((sum, value) => sum + value, __C__);
  • A A: size, B: value < 3, C: 34
  • B A: length, B: value < 3, C: 34
  • C A: size, B: value > 0, C: 42
  • D A: size, B: value < 3, C: 42
Szándékolt helyes válasz: D) A: size, B: value < 3, C: 42 — de figyelem: mivel size esetén a tömb üres, a filter sosem fut le, így a C opció is 42-t ad. A kérdés emiatt kétértelmű (lásd lent).

Magyarázat

A kulcs: Array.from() a length property-t olvassa, NEM a size-t! Tehát {size: 6} egy üres tömböt eredményez (length = undefined → 0).

// Ha A = "size":
Array.from({size: 6}, (v, i) => i)
// → [] (üres tömb, mert nincs .length property)

// Üres tömbön a chain:
[].reverse()          // → []
[].fill(1, 2, 3)      // → []
[].map(v => v - 1)    // → []
[].filter(v => v < 3) // → []
[].reduce((sum, v) => sum + v, 42) // → 42 (initial value visszaadása)

// Eredmény: 42 ✅

Ha A = "length" lenne (opció B), a lánc így működne:

// Ha A = "length":
Array.from({length: 6}, (v, i) => i) // → [0, 1, 2, 3, 4, 5]
  .reverse()                         // → [5, 4, 3, 2, 1, 0]
  .fill(1, 2, 3)                     // → [5, 4, 1, 2, 1, 0]  (index 2-t kitölti 1-gyel)
  .map(v => v - 1)                   // → [4, 3, 0, 1, 0, -1]
  .filter(v => v < 3)                // → [0, 1, 0, -1]
  .reduce((sum, v) => sum + v, 34)   // → 34 + 0 + 1 + 0 + (-1) = 34

// Eredmény: 34 (nem 42!) → ezért opció B nem jó
Trükk kérdés — és egy fontos pontosítás!
  • A opció (size, value < 3, 34): üres tömb → a reduce a kezdőértéket adja vissza → 34.
  • B opció (length, value < 3, 34): ez a "valódi" lánc, de az eredmény 34 (lásd fent), nem 42.
  • C és D opció egyaránt size-t és 42 kezdőértéket használ. Mivel a tömb üres, a filter predikátuma (value > 0 vs. value < 3) sosem fut le — így mindkettő 42-t ad! Szigorúan véve tehát C és D is helyes kimenetet produkál; a teszt szándékolt megoldása a D, de a kérdés kétértelmű.
  • Tanulság: az Array.from() a length property-t olvassa, nem a size-t. A Map/Set használ size-t, az Array.from() nem!
Array.fill() emlékeztető: .fill(value, start, end) — kitölti az indexeket start-tól (inkluzív) end-ig (exkluzív). .fill(1, 2, 3) → csak index 2 kap 1-es értéket.

8. kérdés – Promise chaining: .catch() elhelyezése

JavaScript Promises Error handling

Kérdés

The code below is supposed to:

  • Render user profile, if fetchUserProfile() succeeded
  • Render error message for any error thrown during fetchUserProfile()
  • Ignore any error thrown during trackActivity("fetch_profile")

To meet this criteria, in which line should .catch(() => {}) be placed?

fetchUserProfile() // <-- returns Promise
  .then(profile =>
    trackActivity("fetch_profile") // <-- returns Promise
      // Line A
      .then(() => profile)
      // Line B
  )
  // Line C
  .then(profile => renderProfile(profile))
  // Line D
  .catch(error => renderErrorMessage(error))
  • A Line A
  • B Line B
  • C Line C
  • D Line D
Helyes válasz: A) Line A

Magyarázat

A .catch(() => {})-t a Line A-ba kell helyezni — közvetlenül a trackActivity() UTÁN, de a .then(() => profile) ELŐTT.

fetchUserProfile()
  .then(profile =>
    trackActivity("fetch_profile")
      .catch(() => {})       // ← Line A: trackActivity hibáit elnyeljük
      .then(() => profile)   // majd visszaadjuk a profile-t
  )
  .then(profile => renderProfile(profile))
  .catch(error => renderErrorMessage(error))  // fetchUserProfile hibáit elkapjuk

Miért működik?

// 1. Ha fetchUserProfile() SIKERES + trackActivity() SIKERES:
//    profile → trackActivity → .catch (skip) → profile → renderProfile ✅

// 2. Ha fetchUserProfile() SIKERES + trackActivity() HIBÁT DOB:
//    profile → trackActivity → .catch(() => {}) elnyelődik → profile → renderProfile ✅

// 3. Ha fetchUserProfile() HIBÁT DOB:
//    .then(profile => ...) KIHAGYVA → .catch(error => renderErrorMessage) ✅
Miért nem a többi sor?
  • Line B (a .then(() => profile) után): Túl késő — ha trackActivity hibázik, a .then(() => profile) már nem futna le, és a hiba feljebb buborékolna.
  • Line C (a külső .then előtt): Ez a fetchUserProfile hibáit is elnyelné — a belső .then egész eredményét catch-elné, ami nem jó.
  • Line D (az utolsó .catch előtt): Ez az összes hibát elnyelné (fetchUserProfile + trackActivity), tehát a renderErrorMessage soha nem hívódna meg.
Promise chain szabály: A .catch() mindig az előtte lévő láncszemet kapja el. Ha szűk scope-ban akarsz hibát kezelni, a catch-et közvetlenül az adott Promise után kell elhelyezni, mielőtt a lánc tovább folytatódna.

9. kérdés – Melyik a module bundler?

Tooling JavaScript Web Development

Kérdés

Which of the following is considered a module bundler in JavaScript?

  • A SystemJS
  • B Webpack
  • C CommonJS
  • D RequireJS
Helyes válasz: B) Webpack

Magyarázat

A Webpack az egyetlen igazi module bundler a listában. A bundler az összes modult (JS, CSS, képek) egyetlen vagy néhány fájlba csomagolja a böngésző számára.

// Module bundlerek: összepakolják a fájlokat
Webpack       – a legelterjedtebb bundler (code splitting, tree shaking, loaderek)
Rollup        – ES Module-fókuszú bundler (library-khez ideális)
Parcel        – zero-config bundler
esbuild       – Go-ban írt, ultra gyors bundler
Vite          – dev serverrel kombinált bundler (esbuild + Rollup)

// Module LOADEREK: futásidőben töltik be a modulokat (NEM bundlerek!)
SystemJS      – univerzális module loader (AMD, CommonJS, ES Modules)
RequireJS     – AMD (Asynchronous Module Definition) loader

// Module SPECIFIKÁCIÓK: szabványok/formátumok (NEM eszközök!)
CommonJS      – Node.js modul rendszer (require / module.exports)
AMD           – Asynchronous Module Definition (define / require)
ES Modules    – natív JS modul rendszer (import / export)
Becsapós opciók:
  • SystemJS – Module loader, nem bundler. Futásidőben tölti be a modulokat a böngészőben.
  • CommonJS – Module specifikáció (nem eszköz!). A Node.js require() / module.exports szintaxis.
  • RequireJS – AMD module loader. Aszinkron modul betöltés a böngészőben.

10. kérdés – Érvényes TypeScript változótípusok

TypeScript Types

Kérdés

Which of the following are valid TypeScript variable types?

  • A Array<number>
  • B number[]
  • C Object<number>
  • D number{}
  • E char
  • F [string, number]
  • G any
  • H (foo: number) => string
Helyes válaszok: A, B, F, G, H

Magyarázat

// ✅ ÉRVÉNYES TypeScript típusok:

let a: Array<number>;          // A) Generic array szintaxis
let b: number[];               // B) Shorthand array szintaxis (ugyanaz mint A)
let f: [string, number];       // F) Tuple típus (fix hossz, fix típusok)
let g: any;                    // G) Any típus (bármi lehet)
let h: (foo: number) => string;// H) Függvény típus (number → string)

// ❌ NEM érvényes:

let c: Object<number>;         // C) ❌ Object nem generikus a TS-ben!
                               //    Helyes: Record<string, number> vagy { [key: string]: number }
let d: number{};               // D) ❌ Szintaktikai hiba, nem létező szintaxis
let e: char;                   // E) ❌ Nincs char típus a TS-ben!
                               //    Használd: string (TypeScript nem különböztet char/string között)
Emlékeztető – TS típus szintaxisok:
  • number[] és Array<number> — ugyanaz, tömb
  • [string, number] — tuple (fix szerkezetű tömb)
  • (x: number) => string — függvény típus
  • Record<string, number> — objektum típus (kulcs→érték mapping)
  • any, unknown, never, void — speciális TS típusok

11. kérdés – TypeScript union type deklaráció

TypeScript Union types

Kérdés

Which of the following statements will declare a variable code that can hold either a number or a string?

  • A let code: any;
  • B let code: number | string;
  • C let code: number or string;
  • D let code: string | number;
  • E let code: string or number;
  • F This is not possible in TypeScript.
Helyes válaszok: B, D

Magyarázat

A TypeScript union type szintaxisa a | (pipe) karakter. A sorrend nem számít: number | string és string | number ugyanaz.

// ✅ Helyes union type:
let code: number | string;    // B) ✅
let code: string | number;    // D) ✅ (sorrend nem számít)

// ❌ Nem helyes:
let code: number or string;   // C) ❌ "or" nem TS szintaxis!
let code: string or number;   // E) ❌ ugyanaz a hiba

// ⚠️ Technikailag működik, de NEM helyes válasz:
let code: any;                // A) ⚠️ Elfogad number-t és string-et is,
                              //    DE bármit elfogad (object, boolean, stb.)
                              //    → nem specifikusan "number VAGY string"
Becsapós opciók:
  • A) any – Technikailag "működik" (elfogad number-t és string-et is), de az any MINDENT elfogad. A kérdés specifikusan number VAGY string-et kér, nem "bármit". Az any a type safety teljes feladása.
  • C, E) or kulcsszó – Nem létezik a TypeScript-ben! Nem Java/Python szintaxis. A TS a | (pipe) operátort használja union típusokhoz.
  • F) "nem lehetséges" – Természetesen lehetséges, a union type a TS egyik alapfunkciója.

12. kérdés – TypeScript generic class definíció

TypeScript Generics

Kérdés

What is the correct way to define a generic class in TypeScript?

  • A class List { private data: T[]; }
  • B class List<T> { private data: T[]; }
  • C class <T>List { private data: T[]; }
  • D class List : T { private data: T[]; }
Helyes válasz: B) class List<T> { private data: T[]; }

Magyarázat

A TypeScript generic szintaxisban a típusparaméter a név UTÁN, <szögzárójelben> áll:

// ✅ Helyes generic class szintaxis:
class List<T> {
  private data: T[];

  add(item: T): void {
    this.data.push(item);
  }

  get(index: number): T {
    return this.data[index];
  }
}

// Használat:
const numbers = new List<number>();
numbers.add(42);     // ✅ OK
numbers.add("hello");// ❌ Type error!

const strings = new List<string>();
strings.add("hello");// ✅ OK
Miért nem jó a többi?
  • A) class List { private data: T[]; } – A T nincs deklarálva generic paraméterként → T ismeretlen típusként hibát dob: "Cannot find name 'T'".
  • C) class <T>List – A szögzárójel a név ELŐTT van → szintaktikai hiba. A helyes sorrend: class Név<Paraméter>.
  • D) class List : T – A : más nyelvekben (pl. C#) az öröklődés szintaxisa, TypeScript-ben extends-t használunk. De még extends T sem lenne helyes generic definíció.
Generic szintaxis összefoglaló: 
// Class:
class Box<T> { value: T; }

// Function:
function identity<T>(arg: T): T { return arg; }

// Interface:
interface Pair<K, V> { key: K; value: V; }

// Type alias:
type Result<T> = { data: T; error: string | null; }

13. kérdés – TypeScript type compatibility (strukturális típusosság)

TypeScript Type compatibility Structural typing

Kérdés

In TypeScript, we say that type X is compatible with type Y if an expression of type X can be assigned to a variable of type Y.

Given the following TypeScript definitions:

type A = { amount: number }
interface B { amount: number }
class C { amount: number }
  • A A is compatible with B
  • B B is compatible with A
  • C A is compatible with C
  • D C is compatible with A
  • E B is compatible with C
  • F C is compatible with B
  • G Code above might result in an error, depending on compilation flags
  • H All of the options are correct
Helyes válaszok: A, B, C, D, E, F (mind kompatibilis egymással)
G is igaz (strictPropertyInitialization miatt), H vitatható.

Magyarázat

A TypeScript strukturális típusrendszert (structural typing / duck typing) használ. Ez azt jelenti: ha két típusnak ugyanaz a szerkezete (property-k és típusaik), akkor kompatibilisek – függetlenül attól, hogy type, interface vagy class.

type A = { amount: number }       // type alias
interface B { amount: number }     // interface
class C { amount: number }         // class

// Mindháromnak UGYANAZ a szerkezete: { amount: number }
// Ezért MIND kompatibilisek egymással:

let a: A = { amount: 10 };
let b: B = a;    // ✅ A kompatibilis B-vel (A→B)
let a2: A = b;   // ✅ B kompatibilis A-val (B→A)

let c: C = new C();
c.amount = 10;
let a3: A = c;   // ✅ C kompatibilis A-val (C→A)
let b2: B = c;   // ✅ C kompatibilis B-vel (C→B)

// Fordított irányban is:
let c2: C = { amount: 20 };  // ✅ A struktúra megegyezik
G opció – strictPropertyInitialization:

A class C { amount: number } kód hibát dobhat, ha a strictPropertyInitialization flag be van kapcsolva a tsconfig.json-ban:

// Ha strictPropertyInitialization: true
class C { amount: number }
// ❌ Error: Property 'amount' has no initializer
//    and is not definitely assigned in the constructor.

// Megoldás:
class C { amount: number = 0; }         // alapértékkel
class C { amount!: number; }            // definite assignment assertion
class C { constructor(public amount: number) {} }  // konstruktorban
Strukturális vs Nominális típusosság:
  • TypeScript (strukturális): "Ha úgy néz ki, mint egy kacsa, és úgy hápog, mint egy kacsa, akkor az egy kacsa" — a típus neve nem számít, csak a szerkezet.
  • Java/C# (nominális): "Csak akkor kacsa, ha a Kacsa osztályból származik" — a típus neve/öröklődése számít.

14. kérdés – Interface extends class in TypeScript

TypeScript Interface Inheritance

Kérdés

Select answer(s) that are true regarding the interface extending a class in TypeScript.

  • A Interfaces cannot extend classes
  • B Classes can implement interfaces.
  • C An interface can be implemented by only one class.
  • D Interfaces can extend classes and get members of the class without defining them.
  • E Interfaces can extend only one class.
Helyes válaszok: B, D

Magyarázat

// B) ✅ Classes can implement interfaces:
interface Printable {
  print(): void;
}
class Document implements Printable {
  print() { console.log("Printing..."); }
}

// D) ✅ Interfaces can extend classes and get their members:
class Animal {
  name: string;
  move(distance: number) { /* ... */ }
}
interface Pet extends Animal {
  owner: string;   // Pet-nek van: name, move() + owner
}

// Több interfacet is implementálhat egy class:
class Dog implements Pet {
  name = "Rex";
  owner = "John";
  move(d: number) { /* ... */ }
}
Miért nem jó a többi?
  • A) "Interfaces cannot extend classes" – HAMIS! TypeScript-ben az interface-ek KÉPESEK class-okat extend-álni: interface Pet extends Animal {}.
  • C) "only one class" – HAMIS! Bármennyi class implementálhatja ugyanazt az interface-t. class Cat implements Printable {}, class Dog implements Printable {} — mindkettő ok.
  • E) "extend only one class" – HAMIS! Egy interface több class-t/interface-t is extend-álhat: interface Hybrid extends ClassA, InterfaceB {}.

15. kérdés – TypeScript array union type deklaráció

TypeScript Arrays Union types

Kérdés

Which of the following are correct ways to declare an array arr that can hold any number of elements of any of the following types: string, boolean or number?

  • A let arr: Array<String, Number, Boolean>;
  • B let arr: (string | number | boolean)[];
  • C let arr: string[] | number[] | boolean[];
  • D let arr: [string, number, boolean];
  • E Typescript doesn't support arrays of union types.
Helyes válasz: B) let arr: (string | number | boolean)[];

Magyarázat

// B) ✅ Union type array – HELYES
let arr: (string | number | boolean)[];
arr = ["hello", 42, true, "world", 99];  // ✅ Bármilyen sorrend, bármennyi elem

// Ekvivalens generic szintaxissal:
let arr: Array<string | number | boolean>;  // ugyanaz, egy type paraméterrel
Becsapós opciók:
  • A) Array<String, Number, Boolean> – Array<T> csak EGY type paramétert fogad! Nem adhatunk meg többet vesszővel. Helyes: Array<string | number | boolean>. Plusz: String/Number/Boolean (nagybetűs) a wrapper objektum típusok, nem a primitívek — mindig string/number/boolean (kisbetűs) kell.
  • C) string[] | number[] | boolean[] – Ez NEM kevert tömb! Ez azt jelenti: vagy tiszta string tömb, vagy tiszta number tömb, vagy tiszta boolean tömb. ["hello", 42] nem lenne érvényes, mert egyik sem tiszta típusú.
  • D) [string, number, boolean] – Ez egy tuple: pontosan 3 elem, a megadott sorrendben (1. string, 2. number, 3. boolean). Nem "bármennyi elem".
  • E) – Természetesen támogatja a TypeScript.
Fontos különbség: 
(string | number)[]     // → kevert tömb: ["hello", 42, "world", 99] ✅
string[] | number[]     // → vagy tiszta string[], vagy tiszta number[]
[string, number]        // → tuple: pontosan ["hello", 42], 2 elem, fix sorrend

16. kérdés – TypeScript Point3D: interface extend és intersection type

TypeScript Interface extension Intersection types

Kérdés

Given the following interface:

interface Point {
  x: number; y: number;
}

Which of the following are proper ways to create an interface/type Point3D which would include all properties of Point and additionally z: number, so Point3D will be accepted in all places expecting Point?

  • A type Point3D = Point<{ z: number }>
  • B type Point3D = Record<"x" | "y" | "z", number>
  • C type Point3D = Point & { z: number }
  • D interface Point3D extends Point { z: number; }
  • E interface Point3D implements Point { z: number; }
Helyes válaszok: C, D (és B strukturálisan)

Magyarázat

// C) ✅ Intersection type – a leggyakoribb "type" megoldás:
type Point3D = Point & { z: number };
// Point3D = { x: number; y: number; z: number }
// Az & (intersection) összeolvasztja a két típust

// D) ✅ Interface extends – a klasszikus OOP megoldás:
interface Point3D extends Point {
  z: number;
}
// Point3D örökli az x, y property-ket + hozzáad z-t

// B) ✅ Record utility type – strukturálisan helyes:
type Point3D = Record<"x" | "y" | "z", number>;
// Létrehozza: { x: number; y: number; z: number }
// Strukturálisan kompatibilis Point-tal (x, y megvan)
// De: ha Point-hoz új property kerül, Record nem követi automatikusan!
Miért nem jó a többi?
  • A) Point<{ z: number }> – A Point interface NEM generikus! Nincs <T> paramétere, tehát Point<...> szintaktikai hiba.
  • E) interface Point3D implements Point – Az implements kulcsszó class-oknál használatos, nem interface-eknél! Interface-ek extends-et használnak.
Mikor melyiket válaszd?
  • interface extends → ha a típusod mindig egy adott interface-re épül (tiszta, olvasható)
  • type & {} (intersection) → ha rugalmasan kombinálsz típusokat, különösen type alias-okkal
  • Record<> → ha az összes property azonos típusú és előre ismert a kulcskészlet
