2. A delete operátor viselkedése

JavaScript Scope kód-elemzés

Kérdés

Mit ír ki minden console.log, és miért?

firstName = 'John';      // deklaráció NÉLKÜL → implicit globális
const age = 21;

console.log(delete age);
console.log(delete firstName);
console.log(typeof firstName);
console.log(typeof age);
Kimenet: false · true · "undefined" · "number"

Magyarázat

  • delete age → false: az age egy const (lexikális kötés), nem konfigurálható objektum-property. A delete csak objektum-property-kre hat — változót nem töröl (sloppy módban false-t ad).
  • delete firstName → true: a firstName deklaráció nélkül kapott értéket → implicit globális, ami a globális objektum konfigurálható property-je → törölhető.
  • typeof firstName → "undefined": az előző sor törölte, már nem létezik.
  • typeof age → "number": az age megmaradt.
Csapda: 'use strict' módban más a kép — az implicit globális (firstName = 'John') eleve ReferenceError-t dobna, és a delete egy egyszerű változónévre SyntaxError. Ez a snippet tehát sloppy (nem-strict) módot feltételez.

3. this a globális objektumra — milyen függvény?

JavaScript this binding

🔗 A válaszlehetőségek nem voltak a fotókon — a helyes válasz koncepció alapján.

Kérdés

const objOne = {
  firstName: "Martin",
  surName: "Piyal",
  getFullName(): string {
    return this.firstName + this.surName;
  },
};

Ha azt akarod, hogy a this a globális objektumra mutasson, ne az objOne-ra, milyen típusú függvénnyé alakítsd a getFullName-et?

Helyes válasz: arrow function (nyílfüggvény).

Magyarázat

A metódus-rövidítés (getFullName() {…}) esetén objOne.getFullName() híváskor a this az objOne. Az arrow function nem köt saját this-t — a definíció helyének lexikális this-ét örökli. Mivel az objektum a globális/script szkópban van definiálva, az arrow this-e a globális objektum (sloppy módban), nem az objOne.

const objOne = {
  firstName: "Martin",
  getFullName: () => {
    return this.firstName; // this = globális objektum, NEM objOne!
  },
};
Tanulság: sose írj metódust arrow function-ként, ha a this-re szükséged van az objektumon belül — pont ez a leggyakoribb arrow-this hiba.

4. encode(plainText) — futáshossz-kódolás

JavaScript coding

Feladat

Kódold a szöveget az azonos betűk egymás utáni futásainak megszámolásával. Pl. "aaaabcccaaa" → 4×a, 1×b, 3×c, 3×a → "4a1b3c3a". (Bemenet: csak a–z kisbetűk, sosem null, max. 15000 karakter.)

Megoldás

function encode(plainText) {
  let result = '';
  let count = 1;
  for (let i = 1; i <= plainText.length; i++) {
    // ha a következő ugyanaz, növeljük a számlálót...
    if (i < plainText.length && plainText[i] === plainText[i - 1]) {
      count++;
    } else {
      // ...különben lezárjuk az aktuális futást
      result += count + plainText[i - 1];
      count = 1;
    }
  }
  return result;
}

encode('aaaabcccaaa'); // "4a1b3c3a"
encode('aabaa');       // "2a1b2a"
Lényeg: egyetlen végigjárás (O(n)). A ciklus length-ig megy (inkluzív), így az utolsó futás is lezáródik. Edge case: üres string → "".

5. calculateTotalPrice(prices, discount)

JavaScript coding

Feladat

„Summer Sales": a legdrágább termékre discount% kedvezmény jár (csak egyre). Add vissza a teljes vételárat egész számként (lefelé kerekítve). Korlátok: 0 ≤ discount ≤ 100, 0 < ár < 100000, 0 < termékszám < 100.

Megoldás

function calculateTotalPrice(prices, discount) {
  const total = prices.reduce((sum, p) => sum + p, 0);
  const mostExpensive = Math.max(...prices);
  const saved = mostExpensive * (discount / 100);
  return Math.floor(total - saved);
}

calculateTotalPrice([10, 20, 30], 50); // 30 - 15 = 45 → 45
calculateTotalPrice([100], 10);        // 100 - 10 = 90
Buktatók: (1) a kedvezmény csak EGY (a legdrágább) termékre vonatkozik, nem az egészre; (2) Math.floor kell, nem Math.round; (3) ha holtverseny van a legdrágábból, akkor is csak egyszer vonjuk le.

6. searchValue(numbers, value)

JavaScript coding

Feladat

Adj vissza egy 4 elemű tömböt (mind 0 vagy 1): [1] van-e value páros indexen, [2] páratlan indexen, [3] az első félben, [4] a második félben. value többször is előfordulhat (mindet figyelembe kell venni). A numbers hossza mindig páros; mindig definiált, de lehet üres.

Megoldás

function searchValue(numbers, value) {
  const n = numbers.length;
  const half = n / 2;
  let even = 0, odd = 0, first = 0, second = 0;

  for (let i = 0; i < n; i++) {
    if (numbers[i] === value) {
      if (i % 2 === 0) even = 1; else odd = 1;
      if (i < half) first = 1; else second = 1;
    }
  }
  return [even, odd, first, second];
}
Lényeg: egyetlen végigjárás, minden találatnál mind a négy flag-et frissítjük. Üres tömb → [0,0,0,0] (a páros hossz miatt nincs félosztási kétértelműség).

7. solution('Hello you !') — szavak külön sorba

JavaScript coding

Feladat

A print(solution('Hello you !')) a következőt írja ki (szavanként új sor, \n): Hello / you / !. A bemenet mindig nem-null string.

function solution(str) {
  return str.split(' ').join('\n');
}

function solution(str) {

function nextWeek(date) {

15. Adat átadása a komponensfán át, prop-fűzés nélkül

React Context

🔗 A fotón csak a kérdés volt — a válasz koncepció alapján.

Kérdés

What would you use to pass data through the component tree without having to pass props manually to each level?

Helyes válasz: React Context (React.createContext + useContext).

Magyarázat

A Context megszünteti a „prop drilling"-et (props kézi átfűzése minden szinten). Egy Provider teszi elérhetővé az értéket, és bármely mélyebb komponens useContext-tel olvassa.

const ThemeContext = React.createContext('light');

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Toolbar />   {/* nem kell theme propot fűzni végig */}
    </ThemeContext.Provider>
  );
}

function DeepButton() {
  const theme = React.useContext(ThemeContext); // közvetlenül olvas
  return <button className={theme}>OK</button>;
}

17. ListUsers komponens (Q24/27)

React coding Keys

🔗 Két fotó ugyanerről (skeleton + tesztek, illetve egy átmeneti megoldás).

Feladat

Készíts egy ListUsers komponenst, ami a users prop-ot vezetéknév szerint ábécé-sorrendben (kis/nagybetű-érzéketlenül) jeleníti meg. Minden <li> key-je a vezetéknév. Mutasd a darabszámot („Users: N"). Üres/hiányzó lista esetén „Users: 0" és ne legyen <ul>/<li>. A data-testid-ket ne változtasd.

Megoldás

import React from 'react';

function ListUsers({ users = [] }) {
  const hasUsers = users.length > 0;
  // másolat sortolás előtt – a .sort MUTÁL, ne a propot változtassuk!
  const ordered = [...users].sort((a, b) =>
    a.lastName.toLowerCase() < b.lastName.toLowerCase() ? -1 : 1
  );

  return (
    <div>
      <div data-testid="user-count">Users: {users.length}</div>
      {hasUsers && (
        <ul data-testid="user-list">
          {ordered.map(({ firstName, lastName }) => (
            <li key={lastName}>{firstName} {lastName}</li>
          ))}
        </ul>
      )}
    </div>
  );
}

export default ListUsers;
Buktatók: (1) a [...users] másolat fontos — a .sort() helyben mutálja a tömböt, a props-ot nem szabad módosítani; (2) a key={lastName} az <li>-n legyen (a feladat szerint a vezetéknevek egyediek); (3) üres listánál a {hasUsers && …} miatt egyáltalán nem renderelődik <ul>.

19. Visszaállás a C commitra (history A–F)

Git reset

🔗 Két képből párosítva: a commit-history ábra és a parancs-opciók külön fotón.

Kérdés

Commit-lánc: A — B — C — D — E — F, a HEAD az F-en. A branch-et a C-re akarod állítani, és a lokális fájlok is egyezzenek a C-vel. Melyik parancs?

  • A git reset --hard HEAD~~
  • B git reset --hard HEAD~2
  • C git reset --hard HEAD~4
  • D git reset --hard HEAD~3 helyes
Helyes válasz: D — git reset --hard HEAD~3

F = HEAD     (HEAD~0)
E = HEAD~1
D = HEAD~2
C = HEAD~3   ← ide akarunk jutni  → 3 lépés vissza

20. git commit --amend — mi igaz?

Git amend Több helyes

🔗 Két képből párosítva: a kérdés és a checkbox-válaszlap külön fotón. (Több helyes válasz.)

Kérdés

What is true regarding the --amend flag of git commit? (Multiple correct.)

  • A It can modify only the commit message
  • B It can modify the commit message as well as add new files to the same commit helyes
  • C It can modify multiple previous commits
  • D It can modify only the last commit helyes
Helyes válaszok: B és D.

Magyarázat

Az --amend az utolsó commitot írja felül egy újjal: módosíthatja az üzenetet ÉS/VAGY a tartalmat (a staged változásokat hozzáfűzi). Csak a legutolsó commitra hat (több korábbihoz rebase -i kell).

git commit --amend -m "új üzenet"     # csak az üzenet
git add elfelejtett.js
git commit --amend --no-edit          # új fájl az utolsó commitba, üzenet marad
Vigyázat: az --amend új commit-hasht hoz létre → ha már pusholtad, csak --force(-with-lease)-szel tudod felülírni a távolit (history-átírás).

21. Staging ürítése + working dir a legutóbbi commitra

Git reset

🔗 A fotón csak a kérdés (git status) volt — a válasz koncepció alapján.

Kérdés

$ git status
On branch main
Changes to be committed:
        modified:   index.html
Changes not staged for commit:
        modified:   myScripts.js

Melyik parancs távolít el minden fájlt a staging area-ból ÉS állítja a working directory-t a legutóbbi commit állapotára?

Helyes válasz: git reset --hard HEAD (a HEAD elhagyható: git reset --hard).

Magyarázat

  • git reset --hard HEAD → az indexet (staging) ÉS a working directory-t is a HEAD-hez állítja: az index.html kikerül a stagingből, a myScripts.js módosításai is eldobódnak.
  • Összevetés: git reset HEAD (mixed) csak unstage-el, a fájlmódosításokat megtartja; git reset --soft az indexet sem nyúlja.
Destruktív! A --hard visszavonhatatlanul eldobja a nem-commitolt változtatásokat. Csak akkor, ha tényleg el akarod dobni az összes lokális módosítást.
