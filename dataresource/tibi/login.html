Cookie attribútumok – részletes magyarázat

Web Security HTTP
Attribútum Lehetséges értékek Jelentés / Mire jó?
HttpOnly true / nincs Ha be van állítva, a cookie nem olvasható JavaScript-ből (document.cookie). Véd az XSS támadások ellen, a token nem szerezhető meg JS-ből.
Secure true / nincs Ha be van állítva, a cookie csak HTTPS kapcsolaton keresztül küldhető. HTTP-n nem megy át, így véd a lehallgatás ellen (pl. nyilvános wifi-n).
SameSite Strict / Lax / None Strict: csak azonos domainről küldhető (pl. myapp.com → myapp.com), maximális CSRF védelem.
Lax: alapértelmezett, GET navigációknál (pl. linkre kattintás) küldhető, de űrlap POST-nál nem.
None: minden kérésnél küldhető (cross-site is), de csak Secure mellett!
CSRF támadások ellen véd.
Path pl. /, /api, /admin Meghatározza, hogy a cookie mely útvonalakon lesz elküldve. Pl. Path=/api csak /api alatt érvényes.
Domain pl. .myapp.com Meghatározza, hogy a cookie mely domaineken érvényes. Ha nincs, csak az aktuális domainen.
Expires / Max-Age Expires: dátum
Max-Age: másodperc		 Cookie lejárati ideje. Expires: konkrét dátum/idő. Max-Age: hány másodpercig él. Ha egyik sincs, session cookie (böngésző bezárásáig él).
Priority Low / Medium / High Cookie törlési prioritás böngészőben (ha túl sok cookie van, először a Low törlődik).
Partitioned true / nincs Csak a partitioned storage-ban elérhető (3rd-party cookie privacy védelme, újabb böngészőkben).
__Secure- prefix cookie név elején Csak akkor engedélyezett, ha Secure be van állítva. Pl. Set-Cookie: __Secure-id=123; Secure
__Host- prefix cookie név elején Csak akkor engedélyezett, ha Secure be van állítva, Path=/, Domain nincs megadva. Pl. Set-Cookie: __Host-id=123; Secure; Path=/
Példák:
Set-Cookie: refresh_token=abc123; HttpOnly; Secure; SameSite=Strict; Path=/api; Expires=Wed, 09 Jun 2027 10:18:14 GMT; Priority=High
Set-Cookie: __Secure-id=xyz; Secure; HttpOnly; SameSite=Lax
Set-Cookie: __Host-session=abc; Secure; Path=/; HttpOnly; SameSite=Strict
  • HttpOnly: a cookie-t nem tudja JS ellopni (XSS ellen véd)
  • Secure: csak HTTPS-en megy át (pl. wifi sniffing ellen véd)
  • SameSite=Strict: csak saját domainről küldhető (CSRF ellen véd)
  • Path: csak adott útvonalon érvényes
  • Domain: több subdomainen is érvényes lehet
  • Expires/Max-Age: automatikus lejárat
  • Priority: böngésző törlési sorrend
  • Partitioned: privacy védett, 3rd-party cookie
  • __Secure-: csak Secure mellett, extra védettség
  • __Host-: csak Secure + Path=/ + nincs Domain, legszigorúbb

1. Session vs Token alapok

Autentikáció Backend

Három különböző megközelítés:

A) Szerver-oldali Session (hagyományos)

Hogyan működik: A szerver létrehoz egy session-t, tárolja a memóriában (vagy Redis-ben), és egy session ID-t küld vissza a kliensnek cookie-ban.


USER LOGIN FLOW:
1. POST /login { email, password }
2. Server: password hash ellenőrzés
3. Server: session létrehozása (id: "abc123xyz")
   Server memória: {
     "abc123xyz": {
       user_id: 42,
       email: "user@example.com",
       logged_in_at: 1707036000000,
       ip_address: "203.0.113.42",
       user_agent: "Mozilla/5.0..."
     }
   }
4. Response: Set-Cookie: session_id=abc123xyz; HttpOnly; Secure; SameSite=Strict
5. Kliens: Böngésző automatikusan elküldi a session_id cookie-t minden kérésben

KLIENS KÉRÉS:
GET /api/profile
Cookie: session_id=abc123xyz

SERVER VÁLASZ:
200 OK
{
  user_id: 42,
  email: "user@example.com"
}
Előny Hátrány
Egyszerű (tanár/diák felség) Server must store all sessions
Server-oldali logout (azonnal érvénytelen) Skálázási probléma (több szerver = session sync szükséges)
Visszavonás könnyű Session fixation támadás lehetséges
CSRF megoldható SameSite cookie-val Cross-domain hozzáférés nehézkes

B) Token-based (JWT)

Hogyan működik: A szerver egy aláírt tokent ad vissza. A token tartalmazza az összes szükséges információt (user ID, permissions), és a szerver NEM tárol semmit – csak ellenőrzi az aláírást.


USER LOGIN FLOW:
1. POST /login { email, password }
2. Server: password hash ellenőrzés
3. Server: JWT token generálása (privát kulccsal aláírva)
   Token tartalma: {
     user_id: 42,
     email: "user@example.com",
     role: "admin",
     iat: 1707036000,    // issued at
     exp: 1707122400     // 24 óra múlva lejár
   }
   Aláírás: HMAC-SHA256(payload, secret_key)
4. Response:
   {
     access_token: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
     expires_in: 86400,
     token_type: "Bearer"
   }
5. Kliens: Tároljuk a token-t (localStorage vagy memory)

KLIENS KÉRÉS:
GET /api/profile
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

SERVER VÁLASZ:
1. Token aláírás ellenőrzése (secret_key-vel)
2. Exp dátum ellenőrzése
3. User ID kivonása a tokenből → profil adatok visszaadása
200 OK
{
  user_id: 42,
  email: "user@example.com"
}
Előny Hátrány
Stateless (szerver nem tárol session-okat) Logout = token még érvényes (blacklist szükséges)
Skálázható (minden szerver tudja ellenőrizni) Token meglopás = hozzáférés mindaddig, amíg érvényes
Cross-domain / CORS támogatás Token visszavonás nehézkes
Mobil-friendly Nagyobb payload (minden token tartalmaz adatot)

C) OAuth2 (harmadik féltől jövő token)

Hogyan működik: A felhasználó autentikálódik a Google/GitHub-nál, ők egy token-t adnak vissza, amit a mi szerver elfogad.


OAUTH2 AUTHORIZATION CODE FLOW:
1. Felhasználó: "Bejelentkezés Google-lal"
2. Böngésző → Google OAuth endpoint:
   GET https://accounts.google.com/o/oauth2/v2/auth?
     client_id=...
     redirect_uri=https://myapp.com/callback
     scope=openid profile email
     state=random_string_123
3. Google: Felhasználó bejelentkezik Google-ba
4. Google → Böngésző → myapp.com/callback?code=auth_code_xyz&state=random_string_123
5. Böngésző (kliens) NEM látja az auth_code-ot! A szerver backendje kapja!
6. Backend: OAuth szerver felé küld:
   POST https://oauth2.googleapis.com/token
   {
     code: "auth_code_xyz",
     client_id: "...",
     client_secret: "...",     // TITOK, kliens nem látja!
     redirect_uri: "https://myapp.com/callback"
   }
7. Google: ID token + Access token visszaadása
8. Backend: Felhasználó adatok kinyerése, saját JWT/session létrehozása
9. Backend → Böngésző: JWT token cookie-ban vagy localStorage-ben
10. Kliens: Mostantól a saját JWT token-nel dolgozik (Google token-nel nem kell tovább foglalkozni)
Az OAuth2 fő pontja: Soha ne küldj client_secret-et a böngészőnek! A Google kódot ad vissza, amit a backended felhasznál a secret-tel együtt, hogy megkapja a valódi tokent.

Összehasonlítás:

Aspektus Session JWT OAuth2
Adat tárolása Szerver memória/Redis Token-ben (kliens) Google/GitHub szerveren
Skálázás Nehéz (session sync) Könnyű (stateless) Könnyű (stateless)
CSRF SameSite cookie X-CSRF-Token header State paraméter
Logout Szerver: session törlés Blacklist szükséges Szerver: token blacklist
Mobil támogatás Cookie-s mobilakon nehézkes Tökéletes (header-ben) Tökéletes

2. JWT Token Struktura

JWT Token

JWT formátum: három pont-tal elválasztott rész

Egy JWT három, ponttal elválasztott Base64-szegmens: HEADER (algoritmus), PAYLOAD (claims) és SIGNATURE (a secret kulccsal számolt aláírás).

1. HEADER (Base64 kódolt JSON)


Eredeti:
{
  "alg": "HS256",    // Aláírási algoritmus: HMAC SHA256
  "typ": "JWT"       // Token típus
}

Base64 Encoded:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

2. PAYLOAD (Base64 kódolt JSON + claims)


Eredeti:
{
  "user_id": 42,
  "email": "anna@example.com",
  "role": "admin",
  "permissions": ["read:posts", "write:posts", "delete:posts"],
  "iat": 1707036000,      // Issued At (UNIX timestamp)
  "exp": 1707122400,      // Expires (24 óra múlva)
  "nbf": 1707036000,      // Not Before (opcionális)
  "jti": "unique_id_123"  // JWT ID (opcionális, logout-hoz)
}

Base64 Encoded:
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ

Claims magyarázata:
- user_id, email, role: Custom claims (amit TE adhatsz hozzá)
- iat, exp, nbf: Standard claims (RFC 7519)
- jti: Token azonosítás (logout/revocation-hoz szükséges)

3. SIGNATURE (aláírás a secret kulccsal)


Számítás:
HMAC-SHA256(
  Base64(HEADER) + "." + Base64(PAYLOAD),
  secret_key = "my-super-secret-key"
)

Eredmény:
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Lényeg:
- Csak a SZERVER tudja a secret_key-t
- Ha bárki módosítja a payload-ot (claim-eket), az aláírás érvénytelen lesz
- Ez GARANCIA, hogy a token nem hamis

Kliens-szintű JWT dekódolás:


// JWT token olvasása (aláírás ellenőrzése NÉLKÜL!)
function parseJWT(token) {
  const [headerB64, payloadB64, signatureB64] = token.split('.');

  const payload = JSON.parse(atob(payloadB64));
  // { user_id: 42, email: "anna@example.com", exp: 1707122400, ... }

  return payload;
}

// Kliens-szintű ellenőrzések:
function isTokenValid(token) {
  const payload = parseJWT(token);

  // Ellenőrzés 1: Token lejárt-e?
  const now = Math.floor(Date.now() / 1000);
  if (payload.exp < now) {
    return false; // Lejárt token
  }

  // Ellenőrzés 2: "Not Before" dátum?
  if (payload.nbf && payload.nbf > now) {
    return false; // Még nem érvényes
  }

  return true; // OK, lehet használni
}

// ⚠️ FONTOS: Az aláírás ellenőrzése CSAK a szeveren történhet!
// A kliens nem tudja a secret kulcsot, ezért nem tudja ellenőrizni.
// A szerver MINDIG ellenőrzi az aláírást és az exp dátumot a request-nél.
Gyakori hiba: localStorage vs localStorage-ben tárolás
  • localStorage: Könnyű XSS támadás (malware JS-ben olvassa ki)
  • Cookie (HttpOnly, Secure, SameSite): Védett a JS elől, HTTPS-en küldendő
  • Memory/Redux: Az oldal frissítésén, az token eltűnik (logout-szerű viselkedés)

Ajánlás: JWT token-t HttpOnly cookie-ban tárold, vagy ha localStorage-ben, akkor is kell egy refresh token stratégia.

3. OAuth2 Authorization Code Flow

OAuth2 Autentikáció

A teljes flow lépésről lépésre:

Az Authorization Code flow: a böngésző sosem látja a client_secret-et; az authorization code-ot a backend váltja be valódi tokenre, majd saját JWT-t ad a frontendnek.

Kód például (Node.js + Express + Google OAuth):


// ═══ BACKEND ═══

// npm install googleapis axios

const { google } = require('googleapis');
const axios = require('axios');

const oauth2Client = new google.auth.OAuth2(
  process.env.GOOGLE_CLIENT_ID,
  process.env.GOOGLE_CLIENT_SECRET,
  process.env.GOOGLE_CALLBACK_URL  // https://myapp.com/api/auth/callback
);

// 1. STEP: Felhasználó redirect-je a Google-ra
app.get('/api/auth/google', (req, res) => {
  // State: CSRF védelem (random string)
  const state = Math.random().toString(36).substring(7);
  req.session.oauth_state = state;

  const authUrl = oauth2Client.generateAuthUrl({
    access_type: 'offline',
    scope: ['openid', 'profile', 'email'],
    state: state
  });

  res.redirect(authUrl);
});

// 2. STEP: Google callback – authorization code érkezik
app.get('/api/auth/callback', async (req, res) => {
  const { code, state } = req.query;

  // CSRF védelem: state ellenőrzés
  if (state !== req.session.oauth_state) {
    return res.status(400).json({ error: 'Invalid state parameter' });
  }

  try {
    // 3. STEP: Code → tokens csere (client_secret-tel!)
    const { tokens } = await oauth2Client.getToken(code);
    // tokens: { access_token, id_token, expires_in, ... }

    // 4. STEP: ID token dekódolása
    const ticket = await oauth2Client.verifyIdToken({
      idToken: tokens.id_token,
      audience: process.env.GOOGLE_CLIENT_ID
    });
    const payload = ticket.getPayload();
    // payload: { sub, email, name, picture, ... }

    // 5. STEP: Felhasználó keresése/létrehozása az adatbázisban
    let user = await User.findOne({ google_id: payload.sub });
    if (!user) {
      user = await User.create({
        google_id: payload.sub,
        email: payload.email,
        name: payload.name,
        avatar: payload.picture
      });
    }

    // 6. STEP: Saját JWT token generálása
    const jwtToken = jwt.sign(
      {
        user_id: user.id,
        email: user.email,
        role: user.role
      },
      process.env.JWT_SECRET,
      { expiresIn: '24h' }
    );

    // 7. STEP: Token visszaküldése (session/cookie vagy localStorage)
    res.redirect(`/dashboard?token=${jwtToken}`);
    // VAGY:
    // res.cookie('auth_token', jwtToken, {
    //   httpOnly: true,
    //   secure: true,
    //   sameSite: 'Strict'
    // });
    // res.redirect('/dashboard');

  } catch (error) {
    console.error('OAuth callback error:', error);
    res.status(500).json({ error: 'Authentication failed' });
  }
});

// ═══ FRONTEND ═══

// Bejelentkezés gomb
function LoginWithGoogle() {
  const handleClick = () => {
    window.location.href = '/api/auth/google';
    // Az oauth2Client.generateAuthUrl() eltávolít az accounts.google.com-ra
  };

  return (
    <button onClick={handleClick}>
      Bejelentkezés Google-lal
    </button>
  );
}

// Callback oldalon (myapp.com/callback)
function CallbackPage() {
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const token = params.get('token');

    if (token) {
      // Token tárolása
      localStorage.setItem('auth_token', token);
      // VAGY ha cookie-ban jött: nem kell tenni semmit
      
      // Redirect a dashboard-ra
      window.location.href = '/dashboard';
    }
  }, []);

  return <p>Bejelentkezés folyamatban...</p>;
}
OAuth2 biztonsági pontok:
  • Client Secret: Soha ne küldd a böngészőnek! Csak backend-ben legyen!
  • State paraméter: CSRF védelem – random string, ellenőrzés szükséges
  • Authorization Code: Használati után érvénytelen (egy-szeri kód)
  • Redirect URI whitelist: Google-nál beregisztrálva – csak engedélyezett URL-ek!
  • HTTPS kötelező: Nem lehet HTTP-n OAuth2 (token lopás veszélye)

4. OAuth2 + JWT kapcsolat

OAuth2 JWT Modern Auth

Az alapvető félreértés: nem ugyanaz!

❌ ROSSZ gondolat: "OAuth2 = JWT"
OAuth2 és JWT teljesen különböző dolgok. Az egyik protokoll, a másik token formátum.
✓ HELYES gondolat:
  • OAuth2: Autorizációs protokoll (ki a felhasználó? milyen jogai vannak?)
  • JWT: Token formátum (biztonságos adat csomagolása)
  • Kapcsolat: OAuth2 HASZNÁLHAT JWT-t access tokenként, de nem kötelező

Kettő típusú OAuth2 token kiadás:

A) OAuth2 + Opaque token (tradicionális)

Google, Facebook, GitHub ezeket használ alapértelmezésben:


OAuth2 server (pl. Google) dönt:
↓
ID token: JWT (user info + signature)
Access token: "ya29.a0AfH6SMB..." (opaque string)
                      ↑
                  random karakterek,
                  szerver oldali lookup szükséges

Backend use:
GET https://www.googleapis.com/oauth2/v1/userinfo
Authorization: Bearer ya29.a0AfH6SMB...

Google szerver: lookup az opaque token-ben
                ↓
                { user_id: 12345, email: "..." }

B) OAuth2 + JWT access token (modern, decentralizált)

Sok modern app ezt választja (pl. Auth0, keycloak):


OAuth2 szerver dönt:
↓
ID token: JWT (user info)
Access token: JWT (user info + scope + szignó)
                  ↑
                  Kód által olvasható, kliens-oldal ellenőrizhető

Szignó verifikáció (aszimmetrikus):
- OAuth2 szerver privát kulcssal írja alá
- Kliens publikus kulcsot használhat az ellenőrzéshez
- Backend nem kell a szerver felé query-zni

Backend use:
GET /api/profile
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

Backend: JWT decode + signature verify
         ↓
         { user_id: 42, email: "...", scope: "read" }

Gyakorlati flow: Google OAuth2 + saját JWT

Ez a legjobb kombináció (amit a kódpéldánkban is csináltunk):

A „legjobb kombináció”: Google OAuth2 azonosítja a felhasználót (1–2), a backend saját JWT-t bocsát ki (3), amit a kliens minden API-híváshoz használ (4–5), a backend pedig csak az aláírást verifikálja (6) — nincs Google felé minden kérésnél lekérdezés.

Miért Oauth2 + saját JWT kombináció?

Szempont Csak OAuth2 OAuth2 + saját JWT
Login Google handle-öli (könnyű) Google handle-öli (könnyű)
API hozzáférés Google token (opaque) Saját JWT (legálisabb)
Backend terhelés Google API lekérdezés minden kérésben Csak JWT verifikáció (gyors)
Skálázhatóság Google API rate limit Korlátlan (csak titkod szükséges)
Felhasználó kijelentkezés Google revoke szükséges Helyi blacklist (gyorsabb)
Jogok (roles/scopes) Google decides Te dönthetsz (flexibilis)
Token refresh Google refresh token Saját refresh token (control)

Kódpélda: OAuth2 + JWT kombináció


// BACKEND: OAuth2 callback után

app.get('/api/auth/callback', async (req, res) => {
  const { code } = req.query;

  // 1. Google token szerzése
  const googleTokens = await getGoogleTokens(code);
  // googleTokens:
  // {
  //   access_token: "ya29...",
  //   id_token: "eyJ...",
  //   expires_in: 3599
  // }

  // 2. Google ID token dekódolása
  const payload = await verifyGoogleIdToken(googleTokens.id_token);
  // payload: { sub, email, name, picture, ... }

  // 3. User keresése/létrehozása
  let user = await User.findOne({ oauth_id: payload.sub });
  if (!user) {
    user = await User.create({
      oauth_id: payload.sub,
      oauth_provider: 'google',
      email: payload.email,
      name: payload.name,
      google_access_token: googleTokens.access_token,  // opaque, tároljuk a backenden
      expires_at: new Date(Date.now() + 3599000)
    });
  }

  // 4. SAJÁT JWT token kibocsátása
  const myJWT = jwt.sign(
    {
      user_id: user.id,
      email: user.email,
      oauth_provider: user.oauth_provider,
      role: user.role
    },
    process.env.JWT_SECRET,
    { expiresIn: '24h' }
  );

  // 5. Kliens response
  res.json({
    token: myJWT,           // ← Ez az amit az app használ
    user: user,
    // Megjegyzés: Google access_token titkos marad a backenden!
  });
});

// FRONTEND: Token tárolása és API hívások

function MyApp() {
  const [token, setToken] = useState(null);

}

Összefoglalás – OAuth2 vs JWT:
  • OAuth2: "Ki vagy?" kérdésre válasz (autentikáció + autorizáció protokoll)
  • JWT: "Mit tudsz?" kérdésre válasz (token formátum – adat + szignó)
  • Kombináció: Google OAuth2-vel felhasználót azonosítasz, majd saját JWT-vel dolgozol az appban
  • Opaque token vs JWT: Opaque (sötét, szerver oldali lookup), JWT (fényes, Client-side ellenőrizés)
  • Best practice: OAuth2 (login) + saját JWT (API access) + refresh token (felhasználó session)

5. Token Refresh Mechanizmus

JWT Session Management

A probléma: rövid élettartamú tokenek

A JWT tokeneket rövidebb élettartammal szokás kiadni (15 perc, 1 óra), hogy ha egy tokent ellopnak, a kár korlátozott. De mi van, ha a felhasználó továbbra is aktív marad az appban, és nem szeretné újra beírni a jelszavát?

Megoldás: Refresh token + Access token kettő

Access Token: Rövid élettartam (15 perc)

  • Minden API kéréshez ezt használjuk
  • Ha meglopják, korlátozott az idő amíg hasznos
  • Memória, localStorage vagy HttpOnly cookie-ban

Refresh Token: Hosszú élettartam (7 nap, 30 nap)

  • CSAK az access token megújítására használható
  • CSAK HttpOnly cookie-ban (JavaScript nem férhet hozzá)
  • Szerver oldali tracking (blacklist/whitelist)

Flow diagram:

A refresh-flow: a lejárt access token 401-et vált ki; a kliens a HttpOnly cookie-ban tárolt refresh tokennel új access tokent kér, majd újrapróbálja az eredeti kérést — mindezt a felhasználó észre sem veszi.

Kód implementáció (Node.js + Express):


// Tokens generálása
function generateTokens(userId) {
  const accessToken = jwt.sign(
    { user_id: userId },
    process.env.JWT_SECRET,
    { expiresIn: '15m' }  // 15 perc
  );

  const refreshToken = jwt.sign(
    { user_id: userId, type: 'refresh' },
    process.env.REFRESH_TOKEN_SECRET,
    { expiresIn: '7d' }   // 7 nap
  );

  return { accessToken, refreshToken };
}

// LOGIN endpoint
app.post('/api/auth/login', async (req, res) => {
  const { email, password } = req.body;

  // Email + password ellenőrzés
  const user = await User.findOne({ email });
  if (!user || !bcrypt.compareSync(password, user.password_hash)) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  const { accessToken, refreshToken } = generateTokens(user.id);

  // Refresh token tárolása az adatbázisban (device tracking-hez)
  await RefreshTokenSession.create({
    user_id: user.id,
    token_hash: hash(refreshToken),      // Soha ne tároljunk egyszerű token-t!
    device_fingerprint: req.headers['user-agent'],
    ip_address: req.ip,
    expires_at: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000),
    active: true
  });

  res.cookie('refresh_token', refreshToken, {
    httpOnly: true,
    secure: true,                   // Csak HTTPS
    sameSite: 'Strict',
    maxAge: 7 * 24 * 60 * 60 * 1000 // 7 nap
  });

  res.json({
    access_token: accessToken,
    expires_in: 900  // 15 perc másodpercben
  });
});

// REFRESH endpoint
app.post('/api/auth/refresh', async (req, res) => {
  const refreshToken = req.cookies.refresh_token;

  if (!refreshToken) {
    return res.status(401).json({ error: 'Refresh token missing' });
  }

  try {
    // Refresh token ellenőrzése
    const decoded = jwt.verify(refreshToken, process.env.REFRESH_TOKEN_SECRET);

    // Adatbázisban ellenőrzés (blacklist?)
    const session = await RefreshTokenSession.findOne({
      user_id: decoded.user_id,
      token_hash: hash(refreshToken),
      active: true,
      expires_at: { $gt: new Date() }
    });

    if (!session) {
      return res.status(401).json({ error: 'Refresh token invalid or revoked' });
    }

    // Új access token
    const accessToken = jwt.sign(
      { user_id: decoded.user_id },
      process.env.JWT_SECRET,
      { expiresIn: '15m' }
    );

    res.json({
      access_token: accessToken,
      expires_in: 900
    });

  } catch (error) {
    res.status(401).json({ error: 'Token refresh failed' });
  }
});

// LOGOUT endpoint
app.post('/api/auth/logout', async (req, res) => {
  const refreshToken = req.cookies.refresh_token;

  if (refreshToken) {
    // Adatbázisból deaktiválás
    await RefreshTokenSession.updateOne(
      { token_hash: hash(refreshToken) },
      { active: false }
    );
  }

  res.clearCookie('refresh_token');
  res.json({ message: 'Logged out' });
});

// API Interceptor – automatikus refresh
import axios from 'axios';

const api = axios.create({ baseURL: '/api' });

// Request interceptor: access token hozzáadása
api.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('access_token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// Response interceptor: 401 esetén refresh + retry
api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;

    // Ha 401 és még nem próbáltuk a refresh-t
    if (error.response.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true;

      try {
        // Refresh token request
        const { data } = await axios.post('/api/auth/refresh');
        
        // Új access token tárolása
        localStorage.setItem('access_token', data.access_token);

        // Eredeti kérés újrapróbálása új token-nel
        originalRequest.headers.Authorization = `Bearer ${data.access_token}`;
        return api(originalRequest);

      } catch (refreshError) {
        // Refresh sikertelen – logout szükséges
        localStorage.removeItem('access_token');
        window.location.href = '/login';
        return Promise.reject(refreshError);
      }
    }

    return Promise.reject(error);
  }
);

}

Token refresh best practices:
  • Access token: 15-30 perc élettartam, memória/localStorage
  • Refresh token: 7-30 nap, HttpOnly cookie (JS nem érhet hozzá)
  • Adatbázis tracking: Refresh token session-öket tárolni (logout, device management)
  • Token rotation: Refresh után új refresh token kiadása (növelt biztonság)
  • Device fingerprint: Azonos device-ről jövő refresh-ek csak, más device = újra login

6. Aktív Session Tartása

Session Management

Hogyan tartsd aktívnak a felhasználó session-jét?

Ha egy felhasználó az app-ban marad (de nem végez kéréseket), a token lejárhat. Három stratégia:

A) Silent refresh (Heartbeat)

Háttérben időnként (pl. 10 percenként) refresh a token-t.


useEffect(() => {
  // Hartbeat: 10 percenként refresh
  const interval = setInterval(async () => {
    try {
      const response = await fetch('/api/auth/refresh', {
        method: 'POST',
        credentials: 'include'  // Refresh cookie elküldése
      });
      if (response.ok) {
        const { access_token } = await response.json();
        localStorage.setItem('access_token', access_token);
      }
    } catch (error) {
      console.error('Silent refresh failed:', error);
    }
  }, 10 * 60 * 1000);  // 10 perc

  return () => clearInterval(interval);
}, []);

B) Activity-based refresh

Felhasználó interakció után (click, keypress) refresh a token-t.


const useActivityRefresh = () => {
  const handleActivity = async () => {
    // Ellenőrizni: közel van-e a token lejárata?
    const token = localStorage.getItem('access_token');
    const payload = parseJWT(token);
    const now = Math.floor(Date.now() / 1000);

    // Ha 2 percen belül lejár: refresh
    if (payload.exp - now < 120) {
      try {
        const response = await fetch('/api/auth/refresh', {
          method: 'POST',
          credentials: 'include'
        });
        const { access_token } = await response.json();
        localStorage.setItem('access_token', access_token);
      } catch (error) {
        console.error('Refresh failed:', error);
      }
    }
  };

  useEffect(() => {
    window.addEventListener('click', handleActivity);
    window.addEventListener('keypress', handleActivity);

    return () => {
      window.removeEventListener('click', handleActivity);
      window.removeEventListener('keypress', handleActivity);
    };
  }, []);
};

C) Sliding window (automatikus hosszabbítás)

Minden kérés után automatikusan hosszabbítod az élettartamot.


// Backend: Refresh token-nél új session időpontot beállítani
app.post('/api/auth/refresh', async (req, res) => {
  const refreshToken = req.cookies.refresh_token;

7. Biztonsági Best Practices

Biztonság

Checklist:

✅ Kötelezően teljesítendő:
  • HTTPS – Soha ne HTTP! Token = jelszó
  • HttpOnly cookie – Refresh token-nek (JS nem érheti el)
  • Secure flag – Cookie csak HTTPS-en küldendő
  • SameSite=Strict – Cookie csak saját domainről küldendő (CSRF védelem)
  • CORS konfiguráció – Csak trusted origin-ek (pl. myapp.com)
  • Password hashing – bcrypt, argon2, NEM MD5/SHA1!
  • Rate limiting – Login: max 5 próbálkozás / 15 perc
  • Token expiry – Access: 15-60 perc, Refresh: 7-30 nap
  • Signature verification – Szerver: minden token-t ellenőrizni!
  • Logout invalidáció – Refresh token adatbázisból törlése/deaktiválása
⚠️ Gyakori hibák:
  • localStorage + JWT: XSS támadás könnyű (malware JS-ben olvassa ki)
  • Nincs aláírás ellenőrzés: Hamis token elfogadása
  • Client-side logout: Token törlése a localStorage-ből, de szerver nem tudja (malware új request-et küldhet)
  • HTTP-n OAuth2: Authorization code és token lopás
  • Refresh token lejárta nélkül: Végtelen hozzáférés, ha token meglopásra kerül
  • Nincs device tracking: Refresh token x device → más gépről nem működhet
  • JWT secret kezelés: Client-side secret (nincs titkosság!)

// ═══ BACKEND (Express) ═══

const express = require('express');
const jwt = require('jsonwebtoken');
const bcrypt = require('bcrypt');
const cookieParser = require('cookie-parser');

const app = express();
app.use(express.json());
app.use(cookieParser());

// Middleware: JWT ellenőrzés
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];

  if (!token) {
    return res.status(401).json({ error: 'No token' });
  }

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.userId = decoded.user_id;
    next();
  } catch (error) {
    return res.status(401).json({ error: 'Invalid token' });
  }
};

// LOGIN
app.post('/api/auth/login', async (req, res) => {
  const { email, password } = req.body;

  const user = await User.findOne({ email });
  if (!user) {
    return res.status(401).json({ error: 'User not found' });
  }

  const isValid = await bcrypt.compare(password, user.password_hash);
  if (!isValid) {
    return res.status(401).json({ error: 'Invalid password' });
  }

  const accessToken = jwt.sign(
    { user_id: user.id, email: user.email },
    process.env.JWT_SECRET,
    { expiresIn: '15m' }
  );

  const refreshToken = jwt.sign(
    { user_id: user.id, type: 'refresh' },
    process.env.REFRESH_TOKEN_SECRET,
    { expiresIn: '7d' }
  );

  // Session tárolása az adatbázisban
  await RefreshTokenSession.create({
    user_id: user.id,
    token_hash: hash(refreshToken),
    expires_at: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  });

  res.cookie('refresh_token', refreshToken, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production',
    sameSite: 'Strict',
    maxAge: 7 * 24 * 60 * 60 * 1000
  });

  res.json({ access_token: accessToken, expires_in: 900 });
});

// REFRESH
app.post('/api/auth/refresh', async (req, res) => {
  const refreshToken = req.cookies.refresh_token;

  if (!refreshToken) {
    return res.status(401).json({ error: 'No refresh token' });
  }

  try {
    const decoded = jwt.verify(refreshToken, process.env.REFRESH_TOKEN_SECRET);

    const session = await RefreshTokenSession.findOne({
      user_id: decoded.user_id,
      token_hash: hash(refreshToken),
      expires_at: { $gt: new Date() }
    });

    if (!session) {
      return res.status(401).json({ error: 'Refresh token expired' });
    }

    const newAccessToken = jwt.sign(
      { user_id: decoded.user_id },
      process.env.JWT_SECRET,
      { expiresIn: '15m' }
    );

    res.json({ access_token: newAccessToken, expires_in: 900 });
  } catch (error) {
    return res.status(401).json({ error: 'Invalid refresh token' });
  }
});

import axios from 'axios';
import { useState, useContext, createContext } from 'react';

const AuthContext = createContext();

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  const login = async (email, password) => {
    const { data } = await axios.post('/api/auth/login', { email, password });
    localStorage.setItem('access_token', data.access_token);
    setUser({ email });
    return data;
  };

  const logout = async () => {
    await axios.post('/api/auth/logout', {}, { withCredentials: true });
    localStorage.removeItem('access_token');
    setUser(null);
  };

  const refreshToken = async () => {
    try {
      const { data } = await axios.post('/api/auth/refresh', {}, { withCredentials: true });
      localStorage.setItem('access_token', data.access_token);
      return true;
    } catch (error) {
      logout();
      return false;
    }
  };

  // API instance interceptor
  axios.interceptors.response.use(
    (response) => response,
    async (error) => {
      if (error.response?.status === 401 && !error.config._retry) {
        error.config._retry = true;
        if (await refreshToken()) {
          return axios(error.config);
        }
      }
      return Promise.reject(error);
    }
  );

  axios.interceptors.request.use((config) => {
    const token = localStorage.getItem('access_token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  });

  return (
    <AuthContext.Provider value={{ user, login, logout, refreshToken }}>
      {children}
    </AuthContext.Provider>
  );
}

function LoginForm() {
  const { login } = useContext(AuthContext);
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');

  const handleSubmit = async (e) => {
    e.preventDefault();
    try {
      await login(email, password);
      window.location.href = '/dashboard';
    } catch (error) {
      alert('Login failed: ' + error.response.data.error);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={email} onChange={(e) => setEmail(e.target.value)} placeholder="Email" />
      <input value={password} onChange={(e) => setPassword(e.target.value)} placeholder="Password" type="password" />
      <button type="submit">Bejelentkezés</button>
    </form>
  );
}

Interjúkérdések – gyakorlat

Interjú

Tipikus kérdések:

K1: Mi a különbség a Session és JWT között?

Session: szerver tárol (memory/Redis), cookie-ban csak ID. JWT: tokenben minden adat, szerver stateless. Session: logout könnyű (szerver törlés). JWT: logout nehéz (blacklist szükséges).

K2: Hogyan védekezel a CSRF támadások ellen a modern loginban?

Cookie alapú session: SameSite=Strict flag. JWT/header: X-CSRF-Token header (custom header nem küldhető cross-origin). OAuth2: State paraméter.

K3: Miért kell a token frissítés?

Access token: rövid élettartam (15 perc), ha ellopják, korlátozott ideig hasznos. Refresh token: hosszú élettartam (7 nap), csak frissítésre használható, HttpOnly cookie-ban (JS nem érheti el).

K4: Hogyan tartod aktívnak a felhasználó session-jét?

Három módszer: 1) Silent refresh (10 percenként heartbeat), 2) Activity-based (user click/keypress után), 3) Sliding window (minden refresh után új 7 napos ablak).

K5: Milyen biztonsági intézkedések kellenek?

HTTPS, HttpOnly cookie, Secure flag, SameSite=Strict, password hashing (bcrypt), rate limiting, token expiry, signature ellenőrzés, logout invalidáció, CORS config.

Interjú tipp: Ne csak a flow-t magyarázd el, hanem konkrét biztonsági okokkal is alátámaszd! Miért HttpOnly? Miért lejárat? Miért refresh token? Az interjúztatók ezt szeretik hallani.
